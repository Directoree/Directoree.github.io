0 回调与通知

回调与通知必须要区分开，否则会造成很大的麻烦。

  • 回调对象是 CALLBACK_OBJECT，而通知对象是 EX_CALLBACK。
  • 枚举通知例程的函数为 PspEnumerateCallback，枚举回调的函数为 ExpEnumerateCallback。

本文分析系统版本：

1.1 创建/移除进程通知

需要说明：

  • PsSetCreateProcessNotifyRoutine 只能记录进程创建或消亡，不能阻止进程的创建。
  • PsSetCreateProcessNotifyRoutineEx 可以阻止进程的创建。
  • 通知回调例程中的 Create 参数：
    • 对于创建进程通知：是在被创建的进程第一个主线程中调用的进程通知。
    • 对于删除进程通知：是在进程最后一个主线程终止时调用的进程通知。

三个函数都是要调用 PspSetCreateProcessNotifyRoutine：

判断使用哪个函数：

PspSetCreateProcessNotifyRoutine：

  1. 首先判断 if ( a2 & 1 ) 测试 bit 0 是否为 1，也就是测试 Remove == 1?。如果是则为将通知函数从通知数组中移除。 如果是移除通知数组的成员：

    • 指定 PspCreateProcessNotifyRoutine[i]，然后调用 ExReferenceCallBackBlock 函数。在该函数内，先将通知函数数组目标元素 EX_FAST_REF.RefCnt -= 1，然后在获取 EX_CALLBACK_ROUTINE_BLOCK。并对 EX_CALLBACK_ROUTINE_BLOCK.RundownProtect 运行保护进行校验检查。最后该函数返回通知函数所在的 EX_CALLBACK_ROUTINE_BLOCK 结构。

    • 然后调用 ExCompareExchangeCallBack(arg1,arg2,arg3) 将目标通知函数从数组中摘除。

    • 最后是 PspCreateProcessNotifyRoutineCount/PspCreateProcessNotifyRoutineExCount -= 1。该全局变量表示数组元素的个数。

      • 前者对应 PsSetCreateProcessNotifyRoutine 创建的通知函数。
      • 后者对应 PsSetCreateProcessNotifyRoutineEx/PsSetCreateProcessNotifyRoutineEx2 创建的通知函数。

  2. 如果是创建通知函数。

    1. 调用 MmVerifyCallbackFunctionCheckFlags 判断通知例程所在模块是否在 PE 头中指定 IMAGE_DLLCHARACTERISTICS_FORCE_INTEGRITY 标志，可通过链接器命令行参数处指定：/integritycheck。如果未启用完整性检查标志，则返回 0xC0000022。

      关于 MmVerifyCallbackFunctionCheckFlags 函数的分析，请看本文 2.1.2 节。

    2. 调用 ExAllocateCallBack 函数创建通知对象 EX_CALLBACK_ROUTINE_BLOCK 结构。

    3. 最后调用 ExCompareExchangeCallBack(arg1,arg2,arg3) 将新创建的 EX_CALLBACK_ROUTINE_BLOCK 通知对象插入到 PsSetCreateProcessNotifyRoutine 数组中中元素为 NULL 的位置。

    4. 对通知数组数量的全局变量进行加 1 操作。PspCreateProcessNotifyRoutineCount/PspCreateProcessNotifyRoutineExCount += 1。

    5. 对 PspNotifyEnableMask 进行操作。具体见 2.1.3。

1.1.2 验证通知例程模块完整性

MmVerifyCallbackFunctionCheckFlags 函数功能：

  1. 首先调用 ExAcquireResourceSharedLite 函数，该函数请求 PsLoadedModuleResource 资源的读访问权限。即根据 PsLoadedModuleResource 加载模块参数，去获得模块资源的共享/独占权限。

  2. 然后调用 MiLookupDataTableEntry 函数取得通知函数所在模块的 KLDR_DATA_TABLE_ENTRY 链。

  3. 对通知函数进行如下验证，才表示通过，返回 TRUE：

    • 通知函数所在模块存在。

    • Flags == 0。

    • KLDR_DATA_TABLE_ENTRY.Flags = 0x20。

该函数主要是判断通知例程所在模块是否在 PE 头中指定 IMAGE_DLLCHARACTERISTICS_FORCE_INTEGRITY 标志，可通过链接器命令行参数处指定：/integritycheck。如果是通过 Visual Studio 自身编译器作为交叉编译工具链，则需在“项目-属性-链接器-命令行”位置添加 /INTEGRITYCHECK 即可。

1.1.3 PspNotifyEnableMask

PspNotifyEnableMask 表示是否开启进程/线程/模块通知回调，此变量不受 PatchGuard 保护：

  • Bit 0：是否开启模块通知。
  • Bit 1：是否开启 PsSetCreateProcessNotifyRoutine 创建的进程通知。
  • Bit 2：是否开启 PsSetCreateProcessNotifyRoutineEx/2 创建的进程通知。
  • Bit 3：是否开启 PsSetCreateThreadNotifyRoutine 创建的线程通知。
  • Bit 4：是否开启 PsSetCreateThreadNotifyRoutineEx 创建的线程通知。

PspNotifyEnableMask 是 ULONG 类型。

在 PspSetCreateProcessNotifyRoutine 函数最后会对进程这两个标志位进行设置。

1.2 进程通知函数

创建进程通知有以下三个例程：

  • PsSetCreateProcessNotifyRoutine
  • PsSetCreateProcessNotifyRoutineEx
  • PsSetCreateProcessNotifyRoutineEx2

每个函数对应的通知例程不同。

CREATE_PROCESS_NOTIFY_ROUTINE

通过函数 PsSetCreateProcessNotifyRoutine 创建的通知例程为 PCREATE_PROCESS_NOTIFY_ROUTINE 类型。

PCREATE_PROCESS_NOTIFY_ROUTINE ：

  • Create：TRUE，当前是创建进程的通知。FALSE，当前是进程结束通知。
  • ProcessId：引发该通知的进程 ID。被创建的进程ID。
  • ParentId：表示 ProcessId 进程的父进程。

通知例程没有返回值，也就是不能通过返回值来改变进程创建或者结束的行为。

需要说明：

  • PsSetCreateProcessNotifyRoutine 只能记录进程创建或消亡，不能阻止进程的创建。
  • PsSetCreateProcessNotifyRoutineEx 可以阻止进程的创建。
  • 通知回调例程中的 Create 参数：
    • 对于创建进程通知：是在被创建的进程第一个主线程中调用的进程通知。
    • 对于删除进程通知：是在进程最后一个主线程终止时调用的进程通知。

注意以下两点，否则会蓝屏：

  1. 调用 pPsLookupProcessByProcessId((HANDLE)ProcessId, &pEPOCESS);，系统会对该进程的引用计数累加1，需要在不使用该结构时，及时调用 ObDereferenceObject(pEPOCESS) 解除引用计数。
  2. A driver must remove any callbacks that it registers before it unloads. 在卸载一个驱动之前，必须卸载该驱动创建的所有通知函数，这个操作可以放在 DriverUnload 来完成。

CREATE_PROCESS_NOTIFY_ROUTINE_EX

通过函数 PsSetCreateProcessNotifyRoutineEx 创建的通知例程为 PCREATE_PROCESS_NOTIFY_ROUTINE_EX 类型。

PCREATE_PROCESS_NOTIFY_ROUTINE_EX ：

  • ProcessId：被创建进程的 ID。
  • PEPROCESS：被创建进程的 EPROCESS。
  • PPS_CREATE_NOTIFY_INFO：如果为 NULL，表示进程正在退出。如果不为 NULL，则用来表示被创建进程的信息。

PsSetCreateProcessNotifyRoutineEx2

函数 PsSetCreateProcessNotifyRoutineEx2 创建的通知例程由 PSCREATEPROCESSNOTIFYTYPE 决定。

如果 NotifyType = PsCreateProcessNotifySubsystems，则 NotifyInformation 为 PCREATE_PROCESS_NOTIFY_ROUTINE_EX。这样就和 PsSetCreateProcessNotifyRoutineEx 一样了。

1.3 调用进程通知

当进程对象被创建时，系统并不会调用进程通知回调，只有当第一个线程插入进程中时才会调用该回调。

当对通知数组 PspCreateProcessNotifyRoutine 进行交叉引用时，会发现第一个是 PspCallProcessNotifyRoutines 函数。实际上触发进程通知例程执行的就是该函数，其功能类似于 ExNotifyCallback 函数。

调用 PspCallProcessNotifyRoutines 函数主要有两个地方：

  • PspInsertThread
  • PspExitProcess

参考《Windows内核回调实现原理与逆向调试分析》上面两个函数调用过程如下：

通过逆向分析，该函数 PspCallProcessNotifyRoutines(PEPROCESS pEprocess, __int64 a2, BOOLEAN Create) 第二个参数暂时还没有逆向出来具体含义。

该函数的功能非常简单：

  1. 在一个最多可循环 63 次的循环里，根据 PspCreateProcessNotifyRoutine[i] 过去到通知例程所在的 EX_CALLBACK_ROUTINE_BLOCK 结构。这个过程由 ExReferenceCallBackBlock 函数来实现。
  2. 只要 PspNotifyEnableMask 的 Bit 1、Bit 2 已经置位，就可以调用数组中每一个不为 NULL 的通知例程。

注意，该函数中还涉及到 ExGetExtensionTable(PspDamExtensionHost)，这与内核扩展机制有关。可以参考：

  • BAM internals
  • Windows内核扩展机制研究
  • Yes, More Callbacks — The Kernel Extension Mechanism

2.2 线程通知函数

创建线程通知函数的 PsSetCreateThreadNotifyRoutine/PsSetCreateThreadNotifyRoutineEx 线程通知函数都是 CREATE_THREAD_NOTIFY_ROUTINE。

    • Create == TRUE，ThreadId 表示创建者线程的 TID，而不是被创建的线程 TID。
    • Create == FALSE，表示当前即将要结束的线程 TID。

    • ThreadId 都是被创建线程 TID。

3 模块通知

3.1 创建/摘除模块通知

当一个内核模块（驱动）或者应用层模块（DLL, EXE）被加载时，会触发一个加载模块的通知，当通知发生时，模块己经被映射到内存中，但是模块内的代 码还没有开始执行。模块在进行卸载时也会发送通知。

创建模块通知函数的函数是 PsSetLoadImageNotifyRoutine，该函数会直接调用 PsSetLoadImageNotifyRoutineEx 函数。摘除模块通知函数的例程是 PsCallImageNotifyRoutines。

PsSetLoadImageNotifyRoutineEx

需要说明：

  1. ExAllocateCallBack、ExCompareExchangeCallBack 函数返回类型都是 EX_CALLBACK_ROUTINE_BLOCK。
  2. 指向模块通知例程的指针是 PspLoadImageNotifyRoutine，每个元素都是 EX_CALLBACK 结构。
  3. 模块通知数组中元素的个数使用 PspLoadImageNotifyRoutine 全局变量表示。
  4. 是否启用模块通知，由 PspNotifyEnableMask 的 Bit 0 是否为 1 来决定。
  5. 每次创建模块通知例程时都会调用 EtwWrite 写入 ETW 事件。

3.4 Bypass Image Load

本节主要是在模块通知例程执行时，去阻止模块的执行，而不是加载。因为调用 PsCallImageNotifyRoutines 前模块已经加载到内存了。阻止模块的加载主要分为：

  • 内核层：.sys 。

    主要有以下方法：只需要找到驱动入口点，并且直接返回 STATUS_ACCESS_DENIED(0xC0000022) 错误码就好。这样，已经加载的驱动就会在执行的时候出错，导致驱动启动失败。

  • 应用层：.dll、.exe。

    由于驱动中 DriverEntry 可以直接返回 NTSTATUS，但是 DLL 入口点函数 DllMain 的返回值并不能决定 DLL 是否成功加载，所以主要有以下两种方法：

    • 使用 MmUnmapViewOfSection 来卸载已加载的模块（由于加载进程模块的时候，系统会有一个内部锁，为了避免死锁。要新创建一个线程进行延迟等待，等待DLL加载完成以后再调用函数进行卸载。）——通过PsSetLoadImageNotifyRoutine学习模块监控与反模块监控。
    • 将已加载的模块 PE 头信息抹掉——监控模块加载并卸载。
    • 常规的拦截 DLL 加载的方法是 Minifilter——监视阻止DLL加载、拦截dll注入、拦截dll加载。

内存映射镜像时：使用 NtCreateFile/NtOpenFile --> NtCreateSection 来创建一个 SEC_IMAGE 的内存区对象，然后调用 NtMapViewOfSection 将视图映射到自己的的地址空间。
