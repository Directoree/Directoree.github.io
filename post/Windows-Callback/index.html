1 CALLBACK_OBJECT

本文是内核层的 WDK 开发，并不是 3 环的开发。

本文分析系统版本：

CALLBACK_OBJECT 是一种内核命名对象（匿名的回调对象是无意义的），内核中使用基于改内核回调对象的回调机制。注意这里的回调机制区别于 NotifyRoutine 回调函数。

回调机制为驱动程序提供了一种通用的方法来发送和接收某类通知，这些通告可以是系统某个部件的状态发生变化而产生的通告，也 可以是开发者自定义的某个条件因为条件成立而产生的通告。

系统已经定义三个回调对象供驱动程序使用，每个回调对象都有一个名称和一组属性，这些属性是在创建对象时定义的：

  • \Callback\SetSystemTime，当系统时间发生变化时，此回调提供通知。
  • \Callback\PowerState，当发生以下情况之一时， \Callback\PowerState 回调对象将提供通知：
    • 系统从交流电源切换到直流电电源，反之亦然。
    • 系统电源策略因用户或应用程序请求而更改。
    • 即将过渡到系统睡眠或关闭状态。 驱动程序可以请求通知，以便可以将代码锁定到内存中，以便于关闭。 在电源管理器发送系统设置-power IRP 之前，会通知回调例程。
  • \Callback\ProcessorAdd，将新处理器添加到系统时， \Callback\ProcessorAdd 回调会提供通知。

回调对象的使用分为以下几个步骤：

  1. 创建回调对象（ExCreateCallback），创建回调对象之前需要先调用 InitializeObjectAttributes 来设置其属性。
  2. 注册回调例程（ExRegisterCallback），将回调函数注册到回调对象上。
  3. 触发回调通知（ExNotifyCallback），指定条件发生时可以调用该函数触发回调函数执行。
  4. 卸载已经注册的回调例程（ExUnregisterCallback ）。
  5. 回调对象解引用（ObDereferenceObject），在删除回调对象之前调用。
  6. 删除回调对象。

1.1 创建回调对象

在创建对象之前，驱动程序调用 InitializeObjectAttributes 宏来设置其属性。注意：IRQL <= APC_LEVEL。

当回调对象的属性初始化好之后，便可以调用 ExCreateCallback 函数进行对象创建。回调对象必须具有一个名称，该名称不能与系统定义的回调的名称想同。不允许使用未命名的回调对象，调用方必须指定对象的名称和属性。

ExCreateCallback 例程可以创建新的回调对象或代表调用方打开现有回调对象。

返回值：如果打开或创建回调对象，将返回 STATUS_SUCCESS。 否则，它将返回 NTSTATUS 错误代码来指示失败的性质。

1.2 注册回调例程

回调对象创建好后，需要调用 ExRegisterCallback 函数来将回调对象与回调例程进行绑定。注意：IRQL <= APC_LEVEL。

返回值：如果注册成功则返回 PVOID 类型的注册句柄，该句柄应被视为不透明并保留供系统使用（如给 ExUnregisterCallback 使用）。如果注册失败则返回 NULL。

1.3 触发通知

注意：IRQL <= DISPATCH_LEVEL。

ExNotifyCallback 例程会导致指定回调对象上的所有回调例程得到执行。

驱动程序必须在 IRQL <= DISPATCH_LEVEL 调用 ExNotifyCallback；系统必须在驱动程序调用此调用的同一 IRQL 处调用回调例程。

注意：调用 ExNotifyCallback 会使回调对象上的所有回调例程按顺序都得到执行，并不能指定去执行某一个回调例程。

2 注册表回调

本文先不研究注册表对象的创建、初始化等（CmpCreateObjectTypes/CmpInitCallbacks/CmpDeleteKeyObject）。

注册表操作的通知实际意义上不是上一篇文章进程、线程、模块通知类型，因为没有使用到 EX_CALLBACK、EX_CALLBACK_ROUTINE_BLOCK 结构来描述通知例程。所以我将其归为回调类操作。

2.1 创建/删除回调例程

  • 创建注册表回调例程：CmRegisterCallback、CmRegisterCallbackEx。
  • 删除注册表回调例程：CmUnRegisterCallback。

使用 CmRegisterCallback 安装注册表回调函数，从 Vista 开始将使用 CmRegisterCallbackEx 函数（不推荐使用前者），因为参数内容更丰富。本文主要分析 CmRegisterCallbackEx。可以看到：这两个函数所创建的回调例程都是 PEX_CALLBACK_FUNCTION 类型。

返回值：

  • 创建成功：STATUS_SUCCESS。

  • 创建失败：返回的 STATUS 是以下之一。

    返回代码 说明
    STATUS_FLT_INSTANCE_ALTITUDE_COLLISION 调用驱动程序或其他驱动程序已经为指定的高度Altitude注册了 RegistryCallback 例程。
    STATUS_INSUFFICIENT_RESOURCES 尝试分配内存失败。

删除回调例程的函数是 CmUnRegisterCallback：

返回值：

  • 成功：STATUS_SUCCESS。
  • 如果 Cookie 与已安装的注册表回调例程不匹配，则返回 STATUS_INVALID_PARAMETER。
  • 其他 NTSTATUS 错误代码。

REG_NOTIFY_CLASS的值 意义 Argument2结构体

返回值：

  • Vista之前：该函数的返回值为 STATUS_SUCCESS 之外的错误码的时候，则表示系统会拒绝操作相应的注册表。
  • Vista及之后：增加了 STATUS_CALLBACK_BYPASS 返回值，可以使配置管理器停止处理这个注册表请求，并且返回 STATUS_SUCCESS 给调用线程。

注意，对注册表的每种操作都有操作前（Pre）调用回调例程，和操作后（Post）调用回调例程。如

  • RegNtPreOpenKey 表示打开指定注册表项前调用回调例程。
  • RegNtPostOpenKey 表示打开指定注册表项后调用回调例程。

我们一般使用操作前的 REG_NOTIFY_CLASS，方便做拦截。在回调例程中直接返回 STATUS_ACCESS_DENIED 即可。

2.4 总结

  1. 所有的注册表回调例程都放在 CallbackListHead 链表表尾，该链表每一个成员都是 REGISTRY_CALLBACK 结构。

    CallbackListHead 挂着所有注册表回调函数所在的 REGISTRY_CALLBACK 结构：

    图片引自《Lazarus & BYOVD: evil to the Windows core》。

  2. 回调例程是存放在 REGISTRY_CALLBACK 结构中的。

  3. 全局变量 CmpCallbackCookie 在每一次插入一个新的回调例程时都是自增 1。

  4. CmpCallBackCount用来表示 CallbackListHead 链表上回调例程个数的一个全局变量。

  5. 每次对链表 CallbackListHead 进行操作前，都要使用全局变量 CmpCallbackListLock 来对链表进行互斥上锁操作。

3 对象回调

在对内核对象句柄之前、之后的操作，都可以使用对象回调例程来进行监控并阻止。

3.1 创建/删除对象回调例程

创建对象句柄回调例程的函数是 ObRegisterCallbacks ，删除对象句柄回调例程的函数是 ObUnRegisterCallbacks。

  • 参数一： OB_CALLBACK_REGISTRATION 原型如下：

    • OperationRegistration 成员是一个数组，该数组的个数由 OperationRegistrationCount 指定。每个元素都是 OB_OPERATION_REGISTRATION 结构，原型如下：

      成员 含义
      ObjectType 指向触发回调例程的对象类型的指针。 指定以下值之一：
      PsProcessType：用于进程句柄操作
      PsThreadType：用于线程句柄操作
      ExDesktopObjectType：用于桌面句柄操作。此值在 Windows 10 之前的系统中不支持
      Operations OB_OPERATION 结构，表示对句柄做什么操作。指定以下一个或多个标志：
      OB_OPERATION_HANDLE_CREATE：已打开或将要打开一个新的进程，线程或桌面句柄
      OB_OPERATION_HANDLE_DUPLICATE：已复制或将要复制进程，线程或桌面句柄
      PreOperation 指向回调例程。在请求操作发生之前，系统调用此例程。POB_PRE_OPERATION_CALLBACK。
      PostOperation 指向回调例程。在请求操作发生之后，系统调用此例程。POB_POST_OPERATION_CALLBACK。

      POB_PRE_OPERATION_CALLBACK、POB_POST_OPERATION_CALLBACK 回调例程原型都是一样的：

      OperationInformation：可以通过修改这个参数，拒绝某些操作的权限，以达到保护进程、线程的目的。它是 OB_PRE_OPERATION_INFORMATION 结构：

      Parameters 是一个 OB_PRE_OPERATION_PARAMETERS 联合结构：

      DesiredAccess 默认情况下和 OriginalDesiredAccess 权限是一样的，即拥有全部权限。在回调函数中可修改该值，可修改的权限类型分为两类：

      • 进程权限，如 PROCESS_TERMINATE（可让进程终止的权限）
      • 线程权限，如 THREAD_SET_CONTEXT（可修改线程上下文的权限）

  • 参数二：RegistrationHandle。接受注册成功后的 handle，在驱动卸载或反注册时需要调用 ObUnRegisterCallbacks，并把该参数传入。
