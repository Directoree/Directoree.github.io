1 内核对象

1.1 内核对象

首先需要说一下，内核对象的讲解可以参考《Windows内核原理与实现》章节2.5.1，我这里提炼几点之前不太清楚的知识点：

每一个对象都被包含两部分：对象头（_OBJECT_HEADER）和对象体（Body）。所有对象的对象头都具有统一的格式，其定义见 base\ntos\inc\ob.h。

然而_OBJECT_HEADER与Body并不是整个object的全部，实际上在object header前面还有optional headers与pool header，一个完全的windows object应该是这样的：

  • _POOL_HEADER
  • _OBJECT_QUOTA_CHARGES (optional)
  • _OBJECT_HANDLE_DB (optional)
  • _OBJECT_NAME (optional)
  • _OBJECT_CREATOR_INFO (optional)
  • _OBJECT_HEADER
  • body

对象的两种引用方式：

  • 在内核中直接通过对象地址来引用，ObReferenceObjectByPointer使PointerCount计数减1。

  • 通过句柄来引用对象。使用ObpIncrementHandleCount使HandleCount加1，使用ObpDecrementHandleCount使HandleCount减1。

    • PointerCount：指针计数，它记录了内核本身（也包括驱动程序）引用该对象的次数，一般0环使用。

    • HandleCount：句柄计数，它记录了有多少个句柄引用此对象，这些句柄可能出现在不同的进程中，一般3环使用。

可参考：4Windows内核情景分析—内核对象。

1.2 _OBJECT_HEADER

关于_OBJECT_HEADER.Type.DefaultObject需要进行说明，主要参考《Windows内核情景分析第6.3章节》（《Windows内核原理与实现》没说清楚）：

_OBJECT_HEADER.Type.DefaultObject这个字段或者是指针，或者是位移量：

  • 指针： 指向一个默认的、全局的可等待对象ObpDefaultObject。直接可等待对象的DefaultObject == 0，说明就是目标对象本身。
  • 位移量：指向该类对象（数据结构）内部的一个可等待对象（例如_FILE_OBJECT中的事件Event）。

具体的请看《Windows XP 多核同步 3.2.1DefaultObject》

2 进程句柄表

句柄值：

句柄值 含义
0 ActiveThreads = KTHREAD.ApcState.Process.ActiveThreads
- ActiveThreads != 1，结束当前线程。
- ActiveThreads == 1，直接返回，也就是不能结束当前进程的最后一个线程。
在分析NtTerminateThread(IN HANDLE ThreadHandle OPTIONAL,IN NTSTATUS ExitStatus)时逆得的。
-1 代表当前进程
-2 代表当前线程
一个为4倍数的正数 句柄表中的索引
负数 其值的绝对值为System进程中的句柄值，ObpKernelHandleTable，仅限于内核模式的函数可以引用。

2.1 进程句柄表的含义

只有进程这个内核对象才有句柄表，其他内核对象没有句柄表。

进程内核对象A自己创建或者打开的内核对象的句柄在自己的句柄表里，但是A自己的句柄不在自己的句柄表里，在父进程的句柄表里，但是A主线程的句柄值在自己的句柄表里。

  1. 每个进程对象都有一张属于自身进程的句柄表，当这个进程创建或者打开一个内核对象时，将获得一个句柄，通过这个句柄可以访问内核对象。

  2. 句柄表存储在零环，一个进程使用了几个句柄，在该进程的句柄表中就会存储几个句柄。

  3. 句柄存在的目的是为了避免在应用层直接修改内核对象。

  4. 每一个句柄值存储时是8字节，高4字节是句柄值，低4字节是对应内核对象的地址。

  5. 所有的句柄所对应的内核对象，都包含在 _OBJECT_HEADER 中，真正的内核对象保存在 _OBJECT_HEADER +0x018 body 的位置（如ERPOCESS）。

每一个对象都由两部分构成：对象头和对象体。所有对象的对象头都具有统一的格式，其定义见 base\ntos\inc\ob.h。（还可参见Windows内核原理与实现2.5.1）

句柄值和对应内核对象地址的关系：$$Index\ =\ 句柄值/4$$

$$内核对象\_OBJECT\_HEADER地址\ =\ TableCode[句柄值/4]$$

$$内核对象地址\ =\ \_OBJECT\_HEADER\ +\ 0x18$$

2.2 句柄表TableCode

句柄表是一个多层结构，最多有三层，最少有一层。层数由TableCode的低2位的值来判断，低2位为0时为1层，为1时为2层，为2时为3层。

句柄表有几层是动态创建的，先回使用一层的，句柄变多了容不下了，就会动态创建二级句柄表（Windows内核原理与实现3.4）。

  • TableCode结构为：_HANDLE_TABLE。

  • TableCode每一项的结构为：_HANDLE_TABLE_ENTRY，每一项为8字节。

打印0418_Handle_1.exe的句柄表：

得到TableCode = 0xe1e4b000 ，需要注意最后二位（Windows内核原理与实现_WRK）：

  • 00：0，单层句柄表，4KB = 4096bytes/8bytes = 512项。

  • 01：1，二层句柄表，（4KB/4bytes）*（4KB/8bytes） = 1024*512项。

  • 10：2，三层句柄表，1024*1024*512项。

该字段的低两位清零后的地址为句柄表的最高层表的首地址。

2.3 句柄项_HANDLE_TABLE_ENTRY

打印0418_Handle_1.exe的句柄表中句柄值为0x708项（索引 = 0x708/4 = 0x1c2）。

  • TableCode结构为：_HANDLE_TABLE。

  • TableCode每一项的结构为：_HANDLE_TABLE_ENTRY，每一项为8字节。

句柄值：

句柄值 含义
0 ActiveThreads = KTHREAD.ApcState.Process.ActiveThreads
- ActiveThreads != 1，结束当前线程。
- ActiveThreads == 1，直接返回，也就是不能结束当前进程的最后一个线程。
在分析NtTerminateThread(IN HANDLE ThreadHandle OPTIONAL,IN NTSTATUS ExitStatus)时逆得的。
-1 代表当前进程
-2 代表当前线程
一个为4倍数的正数 句柄表中的索引
负数 其值的绝对值为System进程中的句柄值，ObpKernelHandleTable，仅限于内核模式的函数可以引用。

下面对64位拆分解析，主要参考海哥、火哥和文章Windows内核-句柄。

划分 位数 含义
56-64bit 8位 该8位中的高四位始终为0，后四位标志比如写成函数SetHandleInformation()函数是否设定了HANDLE_FLAG_PROTECT_FROM_CLOSE标志位，如果设定了这四位的值为0x2。
32-55bit 24位 通过获得句柄的函数得到的访问掩码，在进程中就是OpenProcess(dwDesiredAccess,BInheritHandle,dwProcessId)中的dwDesiredAccess。
0-31bit 32位 内核对象的首地址，后三位用0填充。
0-3bit 3位 分别表示三种权限句柄通用权限。

需要特别说明的是0-3bit：

类似TableCode，_HANDLE_TABLE_ENTRY每一项的低3位在查地址时也需要清零，低3位由宏OBJ_HANDLE_ATTRIBUTES来定义（base/ntos/ob/obp.h）：

  • bit 0：OBJ_PROTECT_CLOSE，指示关闭该对象时是否产生一个审计事件。海哥说恒等于1。
  • bit 1：OBJ_INHERIT ，表示是否可以被该进程创建的子进程继承。1：可被继承，0：不可被继承。可以在三个地方修改:
    • 在打开句柄时函数的OpenProcess（dwDesiredAccess，BInheritHandle，dwProcessId），中第二个参数的指定。
    • 采用SetHandleInformation(tempHandle, HANDLE_FLAG_INHERIT, HANDLE_FLAG_INHERIT)来增加权限。
    • 直接修改句柄表项的内容。
  • bit 2：OBJ_AUDIT_OBJECT_CLOSE，已经被取缔为表示句柄表项的锁标志了，如果为1表示句柄表项被锁住了。海哥说恒等于0

3 全局句柄表

3.1 全局句柄表PspCidTable

全局句柄表里：只有所有的进程、线程的句柄，没有其他内核对象。
每个进程都会可能有很多线程，在OS中所有的线程TID和所有进程的PID是不会重复的。

全局句柄表：PspCidTable 指向。

全局句柄表：

  1. 所有的进程和线程无论无论是否打开，都在这个表中。
  2. 每个进程和线程都有一个唯一的编号：PID和TID，这两个值其实就是全局句柄表中的索引。
  3. ETHREAD有一个_CLIENT_ID字段包含了线程的唯一ID和线程对应的进程ID。
  4. 全局句柄表中，4是第一个句柄索引值，它被分配给System进程的PID（因为System进程是第一个通过PspCreateProcess函数创建的进程）。0是专门保留给空闲进程的，它并非通过ExCreateHandle函数而获得。

全局句柄表和进程句柄表的相同点与区别：

相同点：

  1. TableCode低2位含义相同，都是一样的采用动态分层，最多支持三层（512*1024*1024项）。

不同点：

  1. 全局句柄表由全局变量PspCidTable 指向，只保存所有进程及线程句柄值。
  2. 内核句柄表（system进程句柄表）由ObpKernelHandleTable指向。
  3. 全局句柄表里面的内核对象地址值为内核对象的Body而非_OBJECT_HEADER。
  4. PspCidTable 中存放的对象是系统中所有的进程线程对象，其索引就是 PID/4 和 TID/4。
  5. PspCidTable 中存放的直接是对象体（EPROCESS 和 ETHREAD），而每个进程私有的句柄表则存放的是对象头（OBJECT_HEADER）。
  6. PspCidTable：是一个独立的句柄表，而每个进程私有的句柄表以一个双链连接起来注意访问对象时要掩掉低三位，每个进程私有的句柄表是双链连接起来的，实际上ZwQuerySystemlnformation枚举系统句柄时就是走的这条双链，隐藏进程的话，这条链也是要断掉的~~在 遍历进程活动链表（ActiveProcessLinks）、DKOM 隐藏进程时，还要把隐藏进程的句柄表从链表中摘去。
