2 寄存器

2.1 通用寄存器（GPRs）

x64 下指针和地址都是 8 字节（64位）。

  • x64 增加了8个通用寄存器：R8~R15，它们都是64位的。
  • 另外还增加了 8 个 128 位 XMM 寄存器。
  • 64 位寄存器与 x86 下的 32 位寄存器兼容，例如 RAX（64位）、EAX（低32）、AX（低16位）、AL（低8位）、AH（8到15位），相应的有R8、R8D、R8W和R8B。
  • 不要在程序中使用如 AH 之类的寄存器，因为在 AMD 的 CPU 上这种用法会与某些指令产生冲突。

通用寄存器（General-Purpose Registers, GPRs），参考 Intel 手册卷 1 3.4.1。

64 位通用寄存器共 16 个（不包含 RIP，Rflags）。在 x64 中关于 RSP、RBP：

  • RSP ：用来进行相对寻址（RIP-relative addressing）来找到堆栈上的参数、局部变量。不像 x86 基于 EBP 寻址。
  • RBP ：仍可做 Base Frame Point，但是常被用来当通用寄存器使用（GPRs）。

2.2 易失/非易失寄存器

易失寄存器是由调用方假想的临时寄存器，并要在调用过程中销毁。 非易失寄存器需要在整个函数调用过程中保留其值，并且一旦使用，则必须由被调用方保存（所以常会看到堆栈中保存 rbp, rdi）。

  • 易变寄存器：RAX, RCX, RDX, R8-R11
  • 非易变寄存器：RBP, RBX, RSI, RDI, R12-R15

2.3 RFLAGS

在 x64 中，将 EFLAGS 扩展成 64 位，但是高 32 位保留。所以对于 x64 来说，RFLAGS 和 EFLAGS 是一样的。

RFLAGS 的位受算术/逻辑运算影响，同时 RFLAGS 的位也影响 CPU 的执行。如 TF 置位时 CPU 单步运行。

具体标识为可看 Intel 手册卷 1 3.4.3 和 EFLAGS寄存器（标志寄存器）。

  • 将 EFLAG 移入和移出堆栈或 EAX 寄存器：LAHF、SAHF、PUSHF、PUSHFD、PUSHFQ、POPF、POPFD 和 POPFQ。
  • 修改 EFLAG 标识位的指令：BT、BTS、BTR 和 BTC。
  • IA-32e 模式下，不允许修改 VM、NT 位（Intel 手册卷3 2.3.1）。

下面罗列一些与算术、逻辑运算相关的位：

标志位 位置 解释
CF（Carry flag） Bit 0 进位标志。负数计算溢出时置位。
mov al, 0FFh ; CF = 0
add al, 1 ; CF = 1
OF（Overflow flag ） Bit 11 溢出标志。操作发生溢出，导致符号改变（仅对有符号数）。
mov cl, 07Fh ; OF = 0
add cl, 1 ; OF = 1
ZF（Zero flag） Bit 6 零标志。当算术或逻辑运算的结果为零时设置该标志。 这也可以使用比较指令进行设置。
SF（Sign flag） Bit 7 负数符号标志。运算结果是负数时置 1。

3 函数操作

函数调用约定：__fastcall。

  • 调用者在堆栈中会给 RCX、RDX、R8、R9 预留 0x20 字节的空间。

堆栈字节对齐：

  • x86：8 字节。
  • x64：16 字节。

x64 prolog 和 epilog：

分配堆栈空间、调用其他函数、保存非易失性寄存器或使用异常处理的每个函数都必须具有 prolog，其地址限制在与相应函数表条目关联的展开数据中进行了描述。 与之对应的是 epilog。有了这两部分，x64 函数和 x86 函数在堆栈平衡的处理上有了较大的区别。x64 的函数内在一开始使用 prolog 来开辟堆栈，在最后时才使用 epilog 来平衡堆栈。而不像是 x86 中的 __cdecl 调用约定那样每次调用完函数立即要平衡堆栈。x64 是在函数最后的 epilog 来平衡堆栈。

3.1 调用约定

x64 仅使用 __fastcall 函数调用约定。

x86 使用 __cdecl、__stdcall、__fastcall、__thiscall。

x64 中不管在函数前面声明哪一种调用约定，编译器都只会只用 __fastcall。

Visual C/C++ 编译器支持下列调用约定。

关键字 堆栈清理 参数传递
__cdecl 调用方 在堆栈上按相反顺序推送参数（从右到左）
__clrcall 不适用 按顺序将参数加载到 CLR 表达式堆栈上（从左到右）
__stdcall 被调用方 在堆栈上按相反顺序推送参数（从右到左）
__fastcall 被调用方 前 4 个参数依此通过 RCX、RDX、R8、R9寄存器传递，其余参数从右到左压入堆栈
__thiscall 被调用方 在堆栈上推送；存储在 ECX 中的 this 指针
__vectorcall 被调用方 存储在寄存器中，然后按相反顺序在堆栈上推送（从右到左）

3.2 堆栈对齐

堆管理器中的分配粒度相对较小：在 32 位系统上是 8 字节，在 64位 系统上是 16 字节。这部分在《x86 内存管理（二）私有内存-共享内存-物理内存》堆管理器章节提到过。

在 x64 中，为了保证 CPU 访问内存的高效性，堆栈必须 16 字节对齐。下一小节 3个参数Release 用实验来进行说明。

    2. 但是对 16 位的寄存器进行操作，是不会影响高位的，不会自动清零。

  2. 立即数的使用。

    立即数的使用，优先使用 32 位寄存器（因为 64 位指令较长）。如操作数位宽不超过 32 位，硬尽量使用 32 位的寄存器。

    参数长度不超过4个字节的时候，分别使用32位的寄存器进行传参，这个是编译器做的处理，长度足够存储数据的情况下，优先使用32位寄存器，因为操作64位寄存器(如MOV指令)的指令占4个字节，多一个48前缀指令，说明操作数是64位的，而32位寄存器只需要3个字节的指令长度《x64的一些特性》

  3. x64 指令限制。

    • push/pop 的操作数只能是 32 位，无法操作 64 位操作数。pushad/popad 在 x64 中就完全不支持。但是可以使用 pushfq/popfq。

    • 如下：

    • 访问 RIP-2GB~RIP+2GB 范围的地址都会使用 RIP-Relative（这里的 RIP 为当前指令的地址）来进行寻址。

      如48:8B05 前缀使用 RIP-Relative（这里的 RIP 为当前指令的地址）：x64 中，如果要访问的内存地址范围在 RIP-2GB~RIP+2GB ，则硬编码会使用相对寻址的方式编码。

    • jmp 硬编码。

      可看 Intel 手册卷1

      EB：jmp rel8，Jump short，RIP = RIP+当前指令长度 +(FF~7F)。

      E9：jmp rel16，在 x64 下不支持。

      E9：jmp rel32，Jump near，RIP = RIP+当前指令长度 +(-2G~2G)。

  4. 64位下的inline Hook、R3 下常用Hook技术。

  5. 在 x86 CPU 上，FS 段寄存器指向线程环境块 (TEB) 和处理器控制区域 (KPCR)，但在 x64 上，GS 寄存器在用户模式下指向 TEB，在内核中指向 KPCR 的新模式。然而，当运行 WOW64 应用程序（即 x64 系统上的 32 位应用程序）时，FS 寄存器继续指向 32 位版本的 TEB。x64上的陷阱帧结构体nt!_KTRAP_FRAME不包含非易失性寄存器的有效内容。如果打算修改非易失性寄存器，X64函数的prolog会保存非易失性寄存器的值。调试器始终可以从堆栈中提取这些非易失性寄存器的保存值，而不必从陷阱帧中获取它们。 在x64上的内核模式调试期间，.trap命令的输出会打印一条注释，突出显示从陷阱中检索到的所有寄存器的值可能不准确的事实，如下所示。此规则有例外，例如，为用户到内核模式转换生成的陷阱帧确实包含所有寄存器的正确值。

4.5 x64地址范围

x64 体系里只实现了48 位的虚拟地址，高16 位被用做符号扩展。这高 16 位必须全是 0 或者全是 1，这种形式的地址被称为 canonical 地址。

处理器会对软件上的 non-canonical 地址形式进行检查，程序中访问了non-canonical 地址会产生 #GP 异常。

4.6 线性地址

在 x86 保护模式下：

逻辑地址：0x23:00400000。0x23 为段选择子，0x00400000 为段偏移。

线性地址：在分页模式下使用，MMU 将逻辑地址可转换为线性地址。由于除了 FS 其余段基址为 0，所以逻辑地址等于线性地址。

虚拟地址：逻辑地址和线性地址都是虚拟地址。

在 x64 下，只有线性地址（虚拟地址）。

  • 实模式、保护模式下的逻辑地址：$segment:offset$。
    • 实模式：$linear\_address = segment << 4 + offset$
    • 保护模式：$linear\_address = segment\_base + offset$
  • x64：$linear\_address = offset$

在 x64 下，除了 FS 与GS 段可以使用非 0 值的 base 外，其余的 ES、 CS、 DS 及 SS 段的 base 值强制为 0 值。因此，实际上线性地址就等于代码中的 offset 值。

5 64 位模式的指令

5.0 REX prefix

REX prefix 是不定值，它的取值范围是：0x40 - 0x4F （共 16 个），高 4 位固定为 4，低 4 位为变化值。

7 6 5 4 3 2 1 0
0 1 0 0 W R X B

低 4 位表述为：

  • REX.W
  • REX.R
  • REX.X
  • REX.B

当默认操作数大小为 32 位是， REX.W = 1，即前缀取值为 0x48 时，则会将操作数大小改为 64 位。

5.1 操作数大小（operand Size）

在 64 位模式下，仅有少数指令的默认操作数是 64 位，大多数指令的默认操作数仍是 32 位的。在默认操作数为 32 位的指令中，如果要将其变成 64 位的操作数必须使用 REX prefix。

在 64 位模式下指令可以使用 16 位的操作数，但是：如果指令默认操作数大小是 64 位时，不能通过指令前缀来将操作数变成 32 位，但是可以变成 16 位。

示例一：默认操作数大小 64 位

在 x64 dbg 调试器下，push 指令默认操作数大小为 64 位，如果要使用 16 位操作数需加前缀 0x66。不支持 32 位操作数。

示例二：默认操作数大小 32 位

在 x64 dbg 调试器下，mov 指令默认操作数大小为 32 位，如果要使用 16 位操作数需加前缀 0x66。使用 64 位操作数需加前缀 0x48。

默认操作数为64 位的指令
少数指令的默认操作数是 64 位的，这些指令分为两大类：

  1. 依赖于 RIP 指针的 near branch（近程分支）指令。
  2. 依赖于 RSP 指针的 stack 操作指令。

5.2 64 位模式无效指令

5.3 寻址地址大小（operand address size）

  • 在 64 位模式下，指令的 default operand address size（默认地址大小）是 64 位的，指的是操作数的寻址大小。
  • 在 64 位模式下，指令的 default operand size（默认操作数大小）为 64 位或 32 位。

指令的默认地址大小同样由 CS.L 和 CS.D 标志决定。
① CS.L = 1 并且 CS.D = 0，默认的地址是 64 位。
② CS.L = 0 并且 CS.D = 1，在 compatibility 模式下，使用 32 位的默认地址。
③ CS.L = 0 并且 CS.D = 0，在 compatibility 模式下，使用 16 位的默认地址。
