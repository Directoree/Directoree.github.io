1 Windows 子系统架构

参考：《Windows内核原理与实现 P603》、《Windows Internals7 P61》、《The Art of Memory Forensics P407》。

Windows 子系统的组成结构以及两个关键功能部件：窗口管理（GUI）和 GDI（Graphics Device Interface，图形设备接口）。

在 PE 文件可选头 _IMAGE_OPTIONAL_HEADER.Subsystem 指出一个 PE 文件是 IMAGE_SUBSYSTEM_WINDOWS_CUI/IMAGE_SUBSYSTEM_WINDOWS_GUI。

系统总体架构如下：

Windows 子系统分为用户模式和内核模式部分。

  • 用户模式：csrss.exe、子系统 DLL。
  • 内核模式：win32k.sys（win32kbase.sys、win32kfull.sys）

1.1 用户部分

用户部分主要包括：环境子系统进程 csrss.exe、子系统 DLL。

一、环境子系统进程 csrss.exe

对于每个会话，环境子系统进程（Csrss.exe）都有一个实例加载三个 DLL（Basesrv.dll、Winsrv.dll、Sxssrv.dll 和 Csrsrv.dll），它们包含下列支持：

  1. 创建/删除进程、线程。
  2. 通过 ExitWindowsEx 函数关闭窗口应用程序。
  3. 包含 .ini 文件配置注册本地映射以实现向后兼容性。
  4. 将某些内核通知消息（例如来自即插即用管理器的消息）作为窗口消息（WM_DEVICECHANGE）发送到 Windows 应用程序。
  5. 对16 位 DOS 虚拟机（VDM）进程的部分支持（仅32位 Windows)。
  6. SxS（Side-by-side）/Fusion和清单文件（manifest）支持。
  7. 提供多种语言支持的函数，提供缓存。

二、子系统 DLL

子系统 DLL（比如 Kemel32.dll、 Advapi32.dll(注册表相关)、 User32.dll 和 Gdi32.dll），将己经文档化的Windows API 函数，转译成 Ntoskrl.exe 和 Win32k.sys 中恰当的且绝大多数未文档化的内核模式系统服务调用。

类似于用户模式调用执行体内核服务一样，Windows 子系统调用如下：最终通过Win32u.dll进入内核

有一个 Windows 体系架构的重构工程称为 MinWin，子系统DLL作为该重构工程的一部分， 现在往往是由一组特定的库构成，这些库实现了API集（API Set)，然后被链接进子系统 DLL 中，并且通过一种特殊的转发机制来解析其中的函数。将会在后面进行讨论。

当应用程序调用子系统DLL中的某个函数时，可能会发生下述三件事情之一：

  1. 直接中子系统 DLL 包揽所有工作，这一类函数总是在用户模式下完成，不涉及处理器模式切换。
  2. 露要调用 Windows 内核一次或多次。在这种情形下，可能存在一次或多次模式切换。
  3. 该函数要求在环境子系统进程（Csrss.exe）中完成某些工作（环境子系统进程运行在用户模式下， 负责维护那些在其控制下运行的客户应用程序的状态)。在这种情况下，该函数通过 ALPC 消息的形式向环境子系统发送客户机/服务器请求，从而让子系统执行某个操作。然后子系统 DLL 等待应答，收到应答之后再返回调用者。

1.2 内核部分

一、图形设备驱动程序

与硬件相关的图形显示器驱动程序、打印机驱动程序和视频微端口驱动程序。

二、内核模块 Win32k.sys

主要包含以下三个部分：

  1. 窗口管理器（window manager）：它控制窗口显示，管理屏幕输出，采集来自键盘、鼠标和其他设备的输入，同时也负责将用户消息传递给应用程序。
  2. 图形设备接口（GDI, Graphics Device Interface）：它是专门针对图形输出设备的 函数库，其中包括线段、文本和图形的绘制函数，以及图形控制函数。
  3. DireetX 功能的包装函数：Windows 对 DirectX 的支持是在另一个内核驱动程序（Dxgkml.sys）中实现的。

三、Windows 10 和 Win32k.sys

基于 Windows 10 的设备的基本窗口管理要求在不同设备上差异很大。例如，运行 Windows 的完整桌面需要所有窗口管理器的功能，例如可调整大小的窗口、所有者窗口、子窗口等。在手机或小型平板电脑上运行的Windows Mobile 10不需要许多这些功能，因为前景中只有一个窗口，并且无法最小化或调整大小等。loT设备也是如此，它们甚至可能根本没有显示器。（基于Windows 10的设备的基本窗口管理要求因相关设备而异。）

由于这些原因，Win32K.sys 的功能已拆分为多个内核模块，因此特定系统上可能不需要所有模块。这通过降低代码的复杂性并消除其许多遗留部分，大大减少了窗口管理器的攻击面。以下是一些示例：

1.3 MinWin

MinWin 工程的核心是将 Windows 模块（不仅内核模块）的依赖关系梳理出来，并在此基础上构建一个最小的操作系统核，并将操作系统分成多层。MinWin 重构的代码包括内核、内存管理、文件系统、TCP/IP、硬件抽象层和其他基本服务，但不包括图形界面的服务重构。

MinWin 工程下引入了 API Set（集）的概念，API Set 为分层结构的各种 DLL。

MinWin 工程与 API Sets（集）息息相关，MinWin 首先集成在 Windows 7 中，在 Windows 8 得到进一步加强，在 Windows 10 中则为 OneCore 的一部分。Minwin 不仅仅是对内核的代码重构，也是对API 的重构。

下面解释 MinWin 工程在用户模式下与 API Sets 的实际运用。

Minwin 采用了一种新的 API 分制模型，将 API 集（API set ）的定义与实现分开。当应用程序调用一个 DLL 中的函数时，它调用的是虚拟 DLL 中的一个函数，而此虚拟 DLL 在一个实际的系统中将被映射到一个真正实现了此函数功能的逻辑 DLL 中。由于引入了虚拟 DLL，因而不同的系统可以以不同的方式将虚拟 DLL 组合到逻辑 DLL 中。

一、API Set 命名约定

  • 名称必须以字符串 api 或 ext-开头。
    • 以 api 开头的名称表示保证存在于所有Windows版本上的 API。
    • 以 ext- 表示 API 开头的名称，这些 API 可能不存在于所有Windows版本。
  • Lx 表示处于第几层。
  • 此名称不区分大小写。

二、apisetschema.dll 执行转发

  1. 文件 apisetschema.dll 位于 %Windir%\System32\apisetschema.dll，主要功能是将属于虚拟 DLL 的 API 集转发到真正实现函数功能的逻辑 DLL 中。此 DLL 没有包含可执行的代码。

    如一个程序调用 CreateFileMappingA 函数，则调用如下：

    api-ms-win-core-memory-l1-1-1.dll 中只是将调用转发至 kernelbase!CreateFileMappingNumaW。

  2. 从 Windows 7 开始，MinWin 工程引入了新的 DLL ——KernelBase.dll。许多 MinWin 魔法都发生在 KernelBase.dll 中。它包含许多从 Kernel32.dll、Advapi32.dll 重构的功能，因此为了与旧程序兼容，他们在那里留下了“转发器”，必要时将调用传递给 KernelBase.dll。

    因此大部分 kernel32.dll 中的函数调用转发至 kernelbase.dll 中的相应函数，而 kernelbase.dll 调用 ntdll.dll 的函数进入 0 环。

这种转发不光发生在应用层，它们也适用于内核模式。NT 内核本身（ntoskrnl）以前只从 HAL 导入，现在从 ext-ms-win-ntos-ksigningpolicy-l1-1-0.dll 之类的 dll 中导入。适用于转发 win32kbase.sys、win32kfull.sys 等。

apisetschema.dll 解析：
  1. 在内核初始化阶段，加载器 winload.exe 调用 winload!OslpLoadAllModules 和 winload!OsLoadImage 来将 DLL 文件 apisetschema.dll 进行加载。
  2. DLL 被加载后，在系统初始化第一阶段。PspInitializeApiSetMap 函数调用 nt!PsLoadedModuleList 函数来获取已加载的 DLL 的 _LDR_DATA_TABLE_ENTRY 结构（通过循环对比 UNICODE_STRING字符：ApiSetSchema.dll）。这与《Windows XP 驱动开发（二）》中使用 TEB-PEB(_PEB_LDR_DATA)-_LDR_DATA_TABLE_ENTRY 类似。
  3. 根据 _LDR_DATA_TABLE_ENTRY.DllBase 解析 PE 文件的节区 _IMAGE_SECTION_HEADER。找到节区名称为 .apiset 的节。
  4. 调用 MmCreateSection 创建内存区对象，然后调用 MmMapViewInSystemSpace 将内存区映射到系统空间。 .apiset 节的内容就复制到系统空间中，最后将 apisetschema.dll 卸载。
  5. 无论何时当一个新的进程启动时，进程管理器把这个内存区对象映射到新进程的地址空间中。
  6. 最后把进程的 PEB 中的 ApiSetMap 域设置为指向该内存区对象被映射后的基地址处。

总结为：内核在系统引导时读入 apisetschema.dll 中的表结构，并且将它映射到每个进程中。可参考Runtime DLL name resolution: ApiSetSchema - Part I。

三、解析 PE 文件的 .apiset 节

.apiset 节的结构如下：

  1. 该节所在的 apisetschema.dll PE 文件不包含可执行代码。.apiset 节数据为多个 API_SET_NAMESPACE 结构，直到最后一个为全 0。在 Windows 10 x86 中该结构大小为 28/0x1C 字节。
  2. API_SET_NAMESPACE.EntryOffset 指向原始的 API 集（虚拟DLL）结构 API_SET_NAMESPACE_ENTRY。在 PE 文件中 EntryOffset 是一个虚拟地址偏移量。
  3. API_SET_NAMESPACE_ENTRY.NameOffset 为API 集（虚拟DLL）的名称。API_SET_NAMESPACE_ENTRY.ValueOffset 指向虚拟 DLL 转发后的逻辑/实际 DLL，为 API_SET_VALUE_ENTRY 结构。
  4. API_SET_VALUE_ENTRY 有可能涉及二次转发（二次转发：第一次从虚拟 DLL 转发至 kernel32.dll，最后再从 kernel32.dll 转发至kernelbase.dll。即 虚拟DLL—>逻辑DLL1—>逻辑DLL2）。
    • 不存在二次转发：API_SET_VALUE_ENTRY.ValueOffset 为逻辑 DLL 名称偏移。
    • 存在二次转发：API_SET_VALUE_ENTRY.NameOffset 为二次转发目的逻辑 DLL 名称偏移。

1.4 MinWin-OneCore-WCOS

文章《OneCore to rule them all: How Windows Everywhere finally happened》解释了 MinWin 的来源：

  1. 早期针对移动端（Phone OS）、游戏引擎/游戏机（Xbox）、PC 端，使用不同的操作系统，开发人员也是独立的部门（Xbox 衍生于 Windows 2000，Windows CE 为单核操作系统）。于是Windows 团队开始了一个成为 MinWin 的项目，该项目将有助于为所有这些不同的需求创建一个共同的平台。
  2. MinWin 可以解决不同组件之间严重相互依赖的问题。
  3. MinWin 的计划是重新打包和重组操作系统，以解开这些相互依存关系，并创建一个更适合快速创新的操作系统，而不会产生连锁反应。这项工作必须以不破坏现有应用程序的方式进行。核心的 Windows API被组织在应用程序使用的 DLL 中。这些 DLL 可以结合许多不同子系统的功能——例如，kernel32.dll 跨越文件处理、创建进程和线程、管理内存、加载其他库、查询系统硬件的某些属性以及许多其他任务。微软需要尊重和保留这种捆绑，同时仍然允许单独的开发。
  • 减少各种组件相互依赖关系
  • 减少操作系统大小
  • 减少不同平台上代码的独立性（使代码更通用）

MinWin 项目煞费苦心地分析了 Windows 中的各种依赖项，并将操作系统分解为一堆组件。这些是严格分层的，以确保低级组件永远不会依赖于高级组件（低级组件更靠近硬件）。依赖性必须严格从高水平流向低水平。这样，高级部件可以安全地修改，而不会破坏较低级别的部件，也不会使这些更改波及更广泛的操作系统。

现在 MinWin 已经发展到 Windows 10 的 OneCore，如下图即为“One Microsoft，OneCore”：

OneCore 的存在，使得 Windows 10 这个主操作系统内核适用于各种设备，从手机、平板电脑、PC、大屏幕显示器甚至Xbox。也就是操作系统的代码是一样的，都构建在同一“核心”操作系统上。

但是，尽管这些现在基于相同的核心，但它们仍然是该核心之上的不同操作系统。微软希望所有 Windows 设备都基于相同的操作系统，并且它正在构建 Windows Core 操作系统成为该操作系统。微软在LinkedIn上的一篇招聘信息表示，Windows Core OS（WCOS）将是“所有新设备共享的操作系统”。—《The Future of Windows: What Are Polaris and Windows Core OS?》

Windows Core OS（简称WCOS）是 Windows 的全新现代版本，是使 Windows 成为真正通用操作系统的巨大进步。简而言之，WCOS 是 Windows 的公分母，在任何设备类型或架构上跨平台工作，可以通过模块化扩展进行增强，以便在必要时提供设备功能和体验。

2 GUI-窗口管理

如下图所示：

  1. 当用户登录到 Windows 中时，操作系统将为该用户建立一个会话。
  2. 在一个会话中，有一个交互式窗口站，可能还有非交互式窗口站。
    • 在交互式窗口站中：通常有三个桌面——登录桌面、默认桌面和屏幕保护桌面。通常我们运行的图形界面应用程序运行在默认桌面上，在它的三个桌面中，任一时刻只有一个是激活的，输入输出设备归激活的桌面所有。
  3. 在每个桌面中，都有一个顶级窗口列表，有系统菜单、最大化/最小化按钮和滚动条等。

Windows 子系统会话有一个 RIT（Raw Input Thread）线程，负责从输入设备读取原始的输入事件，然后生成消息，寄送到正确的线程消息队列中。每个包含用户界面元素的 线程都应该及时地处理这些消息。通常的做法是在一个消息循环中，不停地获取消息，再分发给目标窗口，由目标窗口的窗口过程来响应。

2.1 会话管理器-创建环境子系统

Windows 的会话管理器（smss.exe）是系统中第一个被创建的用户模式进程，它是在 Windows 执行体和内核初始化完成以后被启动的。Smss 进程的任务之一是，启动 Windows 环境子系统进程 csrss.exe 和 Windows 登录进程 winlogon.exe。比外，smss 还通过系统服务 NtSetSystemlnformation，指示内核加载 win32k.sys 模块。Win32k.sys 针对每个会话被加载一次，它没有创建任何一个设备对象，但是创建了驱 动程序对象 Driver/win32k，因此，当其他模块需要引用 win32k 时，可以通过调用对象管理器的 ObReferenceObjectByName 函数获得此驱动程序对象。

系统启动后，SMSS 内部一般仍运行着两个线程：一个是主线程，它调用 NtWaitForMultipleObjects 永久等待 CSRSS 进程 WinLogon 进程，监视其是否意外退出，一旦退出，即触发蓝屏崩溃；另一个线程是 SMSS 的工作线程，它调用 NtWaitForWorkViaWorkerFactory 等待登录会话有关的任务。

系统初始化时会话管理器（smss.exe）执行的主要工作（参考《软件调试第二版 卷2 P68,P89》、《The Rootkit Arsenal Escape and Evasion in the Dark Corners of the System P182》）：

  1. Smss.exe 对下面注册表位置的文件删除和改名任务。HKLM\SYSTEM\CurrentControlSet\Control\Session Manager 下面的键值对 FileRenameOperations。杀毒软件和反安装程序经常使用这个表键来删除难以直接删除的病毒程序或者当时在使用的模块，因为 SMSS启动时，其他用户态进程还没有运行，所以不会因为其他程序使用某个模块而无法删除它。（软件调试第二版 卷2 P90）

  2. 检查并执行注册表 HKLM\SYSTEM\CurrentControlSet\Control\Session Manager 下的 BootExecute 键值对。为 autocheck autochk *，SMSS 得到这个信息后，便会创建 autochk 进程。

  3. 添加监控任务，比如设置系统环境变量。

  4. 加载 Windows 子系统的内核模块 Win32k.sys。加载的模块在注册表 HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\SubSystems\ 下的 KMode。

  5. 加载 “Known’’ DLLs。在 HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\KnownDLLs\ 下。此时的 smss.exe 需要运行在本地 SYSTEM 账户下。

  6. Smss.exe 创建两个会话：Session 0、Session 1。并且每个会话下面有一个 Smss.exe 实例。

  7. 加载子系统用户模式部分（内核模式部分为 win32k.sys）。创建 Windows 子系统服务进程 csrss.exe。Smss.exe 查找在注册表 HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\SubSystems\ 下的 Required 键值对。

    Required 键下有两个值：Debug、Windows。

    • Debug：表示空。
    • Windows：表示 Csrss.exe。

    一旦 Smss.exe 加载并启动 Csrss.exe，会话就能够支持用户模式应用程序可以调用 Windows API 。

  8. Session 0 的 Smss.exe 启动 wininit.exe 进程，Session 1 的 Smss.exe 启动 winlogon.exe 交互式登录管理进程。然后 Smss.exe 将在循环中等待并侦听本地过程调用 （LPC），以响应生成其他子系统、创建新会话或关闭系统。

2.2 窗口站

建立 Windows 子系统后，Smss.exe 会创建另一个关键的进程—WinLogon.exe。WinLogon.exe 首先要做的一个重要工作就是创建 0 号窗口站（WinStat0，交互式窗口站）和三个桌面。但任何时候只有一个桌面是活动的，该桌面称为输入桌面（Input Desktop）。后续应用程序所创建的窗口，一定属于某一个桌面。

  1. WinLogon.exe 调用 win32kbase!NtUserCreateWindowStation 来创建窗口站。
  2. 创建窗口站后，WinLogon.exe 会调用 win32kbase!NtUserCreateDesktop 来创建桌面，该桌面名称为 WinLogon，登录桌面。然后创建一个名为 Default 的桌面供应用程序使用。
  3. 在创建桌面后，WinLogon 会调用 SetActiveDesktop 函数将供自己使用的桌面设置为当前的活动桌面，于是登录桌面便呈现在用户面前。

一、查看窗口站

以管理员权限运行 SysinternalsSuite 的 WinObj 工具，窗口站被注册到对象管理器名字空间的 Windows\WindowStations目录下或Sessionsle\<x>\Windows\WindowStations 目录下，这里 <x> 代表会话 ID，例如0、1等。Windows 10 19044 x64 的窗口站如下：

  • WinSta0：是登录用户的默认交互式窗口站，在 Session 1 中。Windows 中仅该窗口站为交互式的，这意味着它可以接收用户输入并显示用户界面。
  • Service-0x0-<xxx>$：是非交互式窗口站（在后台运行的服务运行于此）。

非交互式窗口站命名方式：Service-0x<yyy>-<xxx>$。这里 xxx 和 yyy 分别是登录会话标识符的高位部分和低位部分。这些窗口站都包含一个 Default 桌面。

  • Service-0x0-3e7$ ：是运行在 SYSTEM 账户下的进程所对应的窗口站；
  • Service-0x0-3e4$：对应 NETWORK SERVICE 账户；
  • Service-0x0-3e5$：窗口站对应 LOCAL SERVICE 账户。

两个重要的概念：

  1. 会话（session），它代表了一个用户登录到系统中以后的执行环境。任何一个进程都属于一个会话。
  2. 会话空间，是指系统虚拟地址空间中特定的地址范围。

属于同一个会话的进程共享这段地址空间所映射的页面；属于不同会话的进程，其地址空间中这一段范围被映射到不同的页面。

在 Windows中，会话是按编号来区分的。在系统控制台登录的用户会话为 Session 0，其他通过远程桌面或终端服务登录到系统中的会话可以是 Session 1、 Session 2 等。每个会话包含一个或多个窗口站，每个窗口站包含一个或多个桌面。

3.2 窗口站（Window Station）

窗口站是和当前进程和会话（Session）相关联的一个内核对象，它包含剪贴板(clipboard)、原子表、一个或多个桌面（Desktop）对象等。

窗口站的主要结构是 tagWINDOWSTATION，位于 win32k 中，没有相关符号导出。与大多数其他窗口对象不同，窗口站（和桌面）是安全的对象，不是窗口类型的用户对象。这意味着它们的分配、管理和释放是与进程、线程、互斥体和注册表等对象是一样的。因此，他们具有 _POOL_HEADER、_OBJECT _HEADER 等对象头。

探测剪切板的监控

许多恶意软件样本会监控剪贴板操作，因此即使用户使用密码管理器，它们也可以窃取凭据。

一种方法是 HOOK SetClipboardData 函数，并在数据放入剪贴板后将其窃取。

SetClipboardData 函数模拟一个 Ctrl+C 操作，以指定的剪贴板格式将数据放在剪贴板上。

uFormat：有 4 4中格式，ASCII（CF_TEXT）、Unicode (CF_UNICODETEXT)、其他的事 OLE 数据格式。

HOOK 的方法修改了系统代码的完整性，非常容易被检测到。

另一种方法是，定期循环地调用 GetClipboardData 函数来偷取剪切板的数据。但是这种方法不太好，因为在调用 GetClipboardData 之前必须调用 Openclipboard，并且在每个会话中，每一个剪切板同一时间只能被某个进程使用。如果定期循环地调用该函数，它可能会导致争用情况，并意外阻止（通过阻止）合法应用程序访问剪贴板。

剪贴板查看器和侦听器

当数据放到剪切板后，微软推荐的立即查看剪切板数据的方法是调用 SetClipboardviewer 或 AddClipboardFormatListener 函数。

原理是：每当剪切板数据发生变化时，他就会发一个 WM_DRAWCLIPBOARD 消息通知应用程序，调用如上两个函数的应用程序就能够侦听到。

可以使用 Volatility 的 wintree 插件来查看有多少个窗口使用了 CLIPBRDWNDCLASS 结构，就表示使用了剪切板。

3.3 桌面（Desktop）

窗口站（和桌面）是安全的内核对象，不是窗口类型的用户对象。窗口站和桌面具有 _POOL_HEADER、_OBJECT _HEADER 等对象头。

桌面是属于窗口站的内核对象。 它包含一个逻辑显示表面和 GUl Objects，如窗口、菜单或挂钩 （Windows, Menus, Hooks ）。 窗口消息只能在同一桌面上的进程之间发送，桌面也是消息挂钩的边界。参考《A crack on the glass》。

桌面是应用程序窗口和用户界面对象的容器。恶意软件以各种方式利用桌面，从在备用桌面中启动应用程序（使当前登录的用户看不到它们），到勒索软件将用户锁定在自己的桌面之外，直到支付费用。

学习桌面相关结构主要是为了：

  • 隐藏桌面：查找用于对登录用户隐藏应用程序的恶意桌面。
  • 检测勒索软件：检测由勒索软件创建的桌面，以有效地将用户锁定在自己的系统之外。
  • 查找隐藏的线程：GUI 子系统定义了一个 tagTHREADINFO 结构，该结构独立于 ETHREAD，但是包含许多线程相关信息。与使用 EPROCESS.ActiveProcessLinks 来隐藏进程类似，当其他方法查看隐藏的线程失败时，可以只用这个结构来查看。

3.4 原子和原子表（Atom）

Atom 原子：是在同一会话中可以在进程之间共享的字符串（进程之间共享信息）。一个进程将字符串作为参数传递给诸如 AddAtom 或 GlobalAddAtom 之类的函数来（显式地）将原子添加到原子表中。这些 API 返回一个整数标识符，其他进程可以使用该标识符来访问字符串。 原子表是一个哈希桶，其中包含整数到字符串的映射。如下参考《Extraordinary String Based Attacks-SMASHING THE ATOM》。

两种类型的原子：字符串、整数（两种类型都由同一个原子表管理）。

  • 字符串类型：在 0xC001 到 0XFFFF 范围内分配一个原子值。将字符串大小限制为 255 字节。使用引用计数以跟踪使用情况。示例：窗口类名称
  • 整数类型：在 1 到 0xBFFF 范围内定义。整数值直接映射到原子值。示例：标准剪贴板格式。

原子表：局部原子表（在每一个应用程序中）、全局原子表，用户原子表（两个系统范围内）。

3.5 窗口（Window）

窗口（Window）是按钮、滚动条、文本/编辑区域等实体的容器。为了记录它们在用户会话中的使用，窗口管理器将这些实体表示为用户对象并维护它自己的句柄表。因此，当应用程序请求对用户实体执行操作时，窗口管理器会提供句柄值，句柄值由句柄管理器映射到内核内存中对应的对象。注意：这里的对象指的是 GUI 对象，而不是执行体的对象。如窗口对象是 win32k!tagWND、菜单对象是 win32k!tagMENU。

GUI对象HEAD

这些 GUI 对象都有一个 HEAD 结构的头部。HEAD 结构保存句柄值的一个复本（h）和一个锁计数 (cLockObj）。如果对象被引用，该计数会增加，当对象不再被特定的组件引用时，该计数会减少。当锁计数为 0 的时候，窗口管理器就认为该对象不再被使用并将其释放。

了解窗口（Window）的主要目的：

  • 进程有效性检查：您可以将窗口名称与应该拥有它们的进程交叉引用。 例如，explorer.exe 始终拥有一个名为 SysFader 的窗口（一个帮助动画的窗口）。 如果您看到一个名为 explorer.exe 的进程没有此窗口，则它可能不是真正的 Windows 资源管理器。
  • 检测反监控软件：许多恶意软件家族扫描由反病毒、监控和其他安全应用程序创建的窗口名称，以用作规避策略。 例如，他们可能会调用 Findwindow 来查找 Process Monitor Sysinternals，然后尝试关闭应用程序。 或者，恶意软件可能会查询与 Wireshark 相关的窗口，如果存在，则简单地拒绝执行。
  • 元数据取证：窗口标题通常包含元信息，例如系统时间、登录用户名、当前查看的网页和部分 cmd.exe 命令。

由于一个桌面上的窗口可能会以各种方式重叠在一起，因此 Windows 子系统为桌面 中的所有窗口定义了一个Z-序（z-order），即深度顺序。在z-序中，最上面的窗口是用户可以看到的，而下面的窗口则极有可能被隐藏起来了；子窗口跟随所属的父窗口一起被移 动。因此，桌面上的窗口按 Z-序形成了一个单链表。当用户操纵或者切换应用程序的窗口 时，它们的Z-序实际上也在作相应的调整。《Windows 内核原理与实现 P615》
