1 对象目录

1.1 对象目录管理

对象分为内核对象、用户对象。用户对象主要用于 GUI 下（没有对象头）。

  • 内核对象：包含对象头、对象体。
    • 无名对象，其他进程不能通过对象名打开对象，对象不可以共享。不命名的对象一般意义不大。
    • 命名对象，可以通过对象名与其他进程共享内核对象。
  • 用户对象：不具有对象头，参见《GUI Subsystem 用户对象》。

在系统中，通过对象目录对命名对象进行组织管理，对象目录是树状结构而非网状结构。

  1. 根节点： 目录对象（_OBJECT_DIRECTORY）。根节点对象名为\。
  2. 中间节点：目录对象（_OBJECT_DIRECTORY）、符号链接对象（_OBJECT_SYMBOLIC_LINK）。
  3. 叶节点：普通内核对象、目录对象（_OBJECT_DIRECTORY）。

说明：根节点是一个数据目录类型，节点名为 “\”，其为 37 个元素的哈希数组，每个元素串起一个单链表队列，同一个链表上的所有节点对象名的 Hash 值相同。

目录对象：可以是根节点、中间节点、叶节点。

主要关注第一个成员，可以看到它是一个含有 37 = NUMBER_HASH_BUCKETS 个成员的指针数组，每个成员都指向 _OBJECT_DIRECTORY_ENTRY 结构。

  • ChainLink：指向下一个 _OBJECT_DIRECTORY_ENTRY，是个单向链表。同一条链表上的所有节点对象名的 Hash 值相同。
  • Object：指向该 Entry 所管理的那个对象，比如 EPROCESS，ETHREAD 等等。
  • HashValue：该 Entry 的 Hash 值

注意：目录项 _OBJECT_DIRECTORY_ENTRY 并非内核对象，但是每一个内核对象插入到对象目录树时，必须有目录项来进行承接才行。

符号链接 _OBJECT_SYMBOLIC_LINK：

其中 LinkTarget 就是其关联的对象名，关于该对象的解析可看 DKOM – Now with Symbolic Links!。

节点遍历：

如果从根节点或某个中间节点开始遍历某个对象，在路径上的每个节点前加 “\” 进行分隔，就形成一个路径名。

  • 绝对路径：根节点\中间节点\叶节点。
  • 相对路径：中间节点\叶节点。

1.2 创建目录对象

创建目录对象使用 ZwCreateDirectoryObject 函数。

  • DesiredAccess：该参数除了可以使用 ACCESS_MASK 值进行定义外，还可以指定以下一个或多个权限：

    ACCESS_MASK标志 访问类型
    DIRECTORY_QUERY 查询
    DIRECTORY_TRAVERSE 名称查找
    DIRECTORY_CREATE_OBJECT 名称创建
    DIRECTORY_CREATE_SUBDIRECTORY 创建子目录
    DIRECTORY_ALL_ACCESS 上述所有类型

  • 使用 ObjectAttributes 结构指针前，必须先调用 InitializeObjectAttributes 对其进行初始化。

该函数核心是通过 ObCreateObject 创建目录对象，然后再用 ObInsertObject 将对象插入到句柄表中。

对象层次目录中插入，查询和删除目录或目录项常用的函数如下：

  • ObpLookupDirectoryEntry：在一个指定的目录中查找一个指定名称的对象。
  • ObpInsertDirectoryEntry：把一个对象插入到一个目录中。
  • ObpDeleteDirectoryEntry：删除指定目录中的指定对象。

2 对象结构

一个内核对象由以下五部分构成：

  1. 对象内存头
  2. 可选信息头
  3. 对象头
  4. 对象体
  5. 对象脚

1.1 POOL_HEADER

在每块系统内存池被分配的时候，都会默认分配该结构（在申请返回的内存前）。系统内存池即内核 System 进程的堆。

POOL_HEADER 结构用来描述每一块已经分配的内核内存块属性。该结构从 Windows 10 1909开始发生了使用上的变化。具体需要研究分页/非分页内存池的分配相关内容。以下文章较详细描述了 POOL_HEADER 的意义和用途：

1.2 可选信息头

目前，可选信息头共有 8 个，理论上已经全部使用，因为 InfoMask 是 UCHAR 类型，共 8 Bit（每个 Bit 对应一种可选信息头）。

特别说明：扩展信息可选头如果存在，就意味着有对象脚存在。_OBJECT_HEADER_EXTENDED_INFO 中有成员指向对象脚结构。

每个可选信息头的大小、位置顺序都是固定的。如何确定一个可选信息头是否存在：

  • OBJECT_HEADER_CREATOR_INFO：由 OBJECT_HEADER.Flags 决定是否存在。
  • 其他 7 个可选信息头：由 OBJECT_HEADER.InfoMask 决定是否存在。

每个可选信息头对应 OBJECT_HEADER.InfoMask 1 Byte 大小数据中的一位掩码（如下表8-17），然后使用 ObpInfoMaskToOffset[InfoMask & (DesiredHeaderBit | (DesiredHeaderBit-1))] 找到期望的可选信息头的偏移，最后用 OBJECT_HEADER 起始地址减去偏移即可得到可选信息头的起始地址。

该公式参考 CodeMachine - Article - Windows Object Headers。该文章指出，这个数组应该足够大，才能容纳 InfoMask(UCHAR) 8 Bits 共 $2^8$ 种可能性偏移。

可选信息头是否存在，是在系统初始化或者对象创建时决定的，存在的条件如表 8-18。

简单总结可选信息头存在的条件：

  1. 名称信息头 _OBJECT_HEADER_NAME_INFO：当一个对象是有名称对象时，该信息头存在。
  2. 配额信息头 _OBJECT_HEADER_QUOTA_INFO：如果对象是被系统初始化进程或空闲系统进程创建的，该信息头存在。
  3. 进程信息头 _OBJECT_HEADER_PROCESS_INFO：如果 _OBJECT_HEADER.ExclusiveObject == 1，则信息头存在。
  4. 句柄信息头 _OBJECT_HEADER_HANDLE_INFO：如果句柄类型的 _OBJECT_TYPE_INITIALIZER.MaintainHandleCount == 1，则信息头存在。
  5. 创建信息头 _OBJECT_HEADER_CREATOR_INFO：如果句柄类型的 _OBJECT_TYPE_INITIALIZER.MaintainTypeList == 1，则信息头存在。
  6. Audit 信息头 _OBJECT_HEADER_AUDIT_INFO：只有文件对象才具有该信息头。
  7. 需要信息脚 OBJECT_FOOTER 的对象，才具有 _OBJECT_HEADER_EXTENDED_INFO 扩展信息头。
  8. 填充信息头 _OBJECT_HEADER_PADDING_INFO：如果句柄类型的 _OBJECT_TYPE_INITIALIZER.CacheAligned == 1，则信息头存在。

1.4 OBJECT_HEADER.Flags

OBJECT_HEADER.Flags 是一些标志位（UCHAR 类型仅 8 位），描述当前对象的属性、访问权限等。创建对象时，Flags 的信息来自于 _OBJECT_ATTRIBUTES。

当通过 Windows 子系统中的 APl（例如 CreateEvent 或 CreateFile）创建对象时，调用者不指定任何对象属性——子系统 DLL 在幕后执行工作。 因此，通过 Win32 创建的所有命名对象都位于 BaseNamedObjects 目录中，无论是全局对象还是每个会话中的对象，因为这个目录是 Kernelbase.dll 指定为对象指定的。 《Windows Internals 7 Part2 Chapter 8》

在创建对象时，先使用 InitializeObjectAttributes 将对象属性进行初始化赋值。其中指定的以 OBJ_ 开头的属性就是传递给 OBJECT_HEADER.Flags 的属性。

3 对象类型

对象是进行分类的，每个对象都有所属的类型。为了做到这一点，每个对象都需要对应到一个类型对象（OBJECT_TYPE）上去，类型对象用来描述该对象所属的类型及其一些属性。对象类型中的描述的属性是该类型下所有对象共有的属性。

系统为每一种对象类型创建了一个内核对象，用该对象来对某一类的对象进行描述，这个对象结构为 OBJECT_TYPE。系统中约有 64 种类型的对象，那就会有 64 个 OBJECT_TYPE 结构存在，每一个 OBJECT_TYPE 结构的 Name 成员指出当前类型对象所描述的对象类型。需要理清楚几个概念：

  1. 系统使用类型对象 OBJECT_TYPE 来将各种对象类型进行分类，对相同类型的对象属性进行定义。
  2. 同一类内核对象对应的 OBJECT_TYPE 都是同一个，如 EPROCESS 1、EPROCESS 2 对应的类型对象 OBJECT_TYPE 是同一个，且 OBJECT_TYPE.Name == Process。
  3. 并不是每个内核对象创立时都会新建一个类型对象。类型对象是在系统初始化时创建好的，后面每当新建内核对象时，都要指定对象对所属的全局类型对象指针（如 PsProcessType）。
  4. 同一类型的内核对象和类型对象 OBJECT_TYPE 的关系是：多对一，如下图所有进程对象对应的类型对象都是同一个。

系统预定义的对象类型约有 64 种，但是系统可以开放出来让开发者自己定义想要的对象类型。

系统中有全局指针指向这个 64 个类型对象（指针类型为 POBJECT_TYPE），在这个类型对象下面的双链表指针串着相同类型的所有对象。如下是 WRK 中列出的 31 个指向类型对象的指针（每个指针类型为 POBJECT_TYPE）。

一、Windows XP

Windows XP 中使用 Type 直接指向 OBJECT_TYPE 结构。

二、Windows 7/8/8.1

Windows 7/8/8.1 下，使用 TypeIndex 索引来取代 Type 成员。

此时的 TypeIndex 是全局数组 ObTypeIndexTable 的索引，该数组中的每个成员都是 POBJECT_TYPE 指针类型，指向其对应的类型对象。
