1 IA-32 和 IA-32e 架构

1.1 IA-32e 架构

  1. 从 32 位系统过渡到 64 位系统初期，Intel 开发了 IA-64，仅支持 64 位指令集，不支持原先的 x86（IA-32）指令集。而 AMD 开发了能支持 IA-32 指令集的 CPU，称之为 x86-64 或 AMD64（简称 x64）。之后由于市场的因素，Intel 跟着 AMD 也在自己的 x86 基础上开发出兼容 IA-32的 CPU，称 IA-32e 或者 Intel64。

    • x86：IA-32

    • AMD64

    • Intel64：IA-32e

  2. IA-32e 模式下支持两种子模式的软件运行。

    • 64-bit mode：Run 64 bit Apps。
    • Compatibility mode（兼容模式）：Run 64 bit Apps，run legacy software。（Legacy Mode：Real mode、Protected mode、Virtual 8086 mode，称为传统模式）

  3. 64 位模式。

    • AMD：Long Mode
    • Intel：IA-32e

  4. 32、64 位。

    • Legacy Mode（传统模式）：内核 32 位，用户 32 位。
    • 64-bit mode：内核 64 位，用户 64 位。
    • Compatibility mode（兼容模式）：内核 64 位，用户 64/32 位。

IA-32e 模式下，系统级的寄存器如下：

  • Model-specific registers（MSR）：用于 R0 代码的寄存器（驱动可访问），用于控制——调试扩展、性能监控计数器、机器检查架构和内存类型范围 (MTRRs) 等。MSR 很复杂，Intel 专门出了手册卷 4 来进行说明。
  • 描述符寄存器：GDTR、IDTR、LDTR 和 TR。
  • 标志寄存器：RFLAGS，64 位中高 32 位保留未使用。R0、R3 都可直接访问和修改（R3仅可修改部分位，如ZF）。
  • 调试寄存器：DR0~DR7，64 位。
  • 控制寄存器：CR0~CR4，CR8，都是 64 位。
  • CPUID 指令，不是寄存器。

开机后，模式启动顺序：实模式 — 保护模式 — IA-32e 模式。

1.2 IA-32e 门的支持

IA-32e 取消了任务门， TSS 还存在（取消了大部分成员），但是系统调用时不会从 TSS 中获取 RSP0。

调用门促进 64 位模式和兼容模式之间的转换。 IA-32e 模式不支持任务门。

支持：调用门、中断门、陷阱门。

最新的手册：Intel® 64 and IA-32 Architectures Software Developer Manuals。

1.3 IA-32e 运行模式

x86 支持三种模式：实模式、保护模式、虚拟8086模式。

Intel 64 可以切换到 x86 的模式中，同时还支持 IA-32e 模式。在 IA-32e 模式下，处理器支持两种子模式（Sub-modes）：

  1. 64-bit 模式（不支持运行32位程序）
  2. Compatibility mode，兼容模式（支持运行32位程序）。

在 IA-32e 下区分 64 位模式和还是兼容模式：查看 CS 段寄存器的 L 位。（代码段描述符增加一个L属性位，用于表明这个是要一个64位代码段）

  • L = 1 : 64位代码段
  • L = 0:
    • D = 1 32位代码段
    • D = 0 16位代码段

注意：兼容模式（Compatibility mode）并不是切换到保护模式，是在 IA-32e 模式下。同时 IA-32e 不支持虚拟 8086 模式。

1.5 激活 IA-32e 模式

操作系统应遵循以下一般顺序来激活 IA-32e 模式：

  1. 从启用分页保护模式开始，通过设置 CRO.PG 三 O 禁用分页。使用 MOV CRO 指令禁用分页（必须位于标识映射页中）。

IA-32e 模式的启用条件是：

　　1. 启用段内存管理进入保护模式 CR0.PE = 1

　　2. 启用PAE内存分页管理模式 CR4.PAE = 1

　　3. 启用64位模式 EFER.LME = 1

　　4. 启用页内存管理 CR0.PG = 1

　　5. 上述四点必须按照上面的顺序启用

　　如果条件全部满足，CPU会置 EFER.LMA = 1表示 IA-32e 位模式启动了 。然后在 IA-32e 下区分 64 位模式和还是兼容模式：查看 CS 段寄存器的 L 位。

1.6 64-bit模式和兼容模式

概括来说：64-bit 模式和兼容模式在 IA-32e 激活后并存。

  • 64-bit 模式用来运行 64 位的程序。
  • 兼容模式，用来运行16/32 位的程序。

兼容模式：IA-32e 模式中的兼容模式保持与传统 IA-32 16 位和 32 位应用程序的二进制兼容性。 （兼容模式不支持以虚拟 8086 模式运行或使用硬件任务管理的旧版 16 位或 32 位应用程序）。 兼容模式是在代码段描述符的基础上选择的。 它允许遗留应用程序在 64 位操作系统下与在 64 位模式下运行的 64 位应用程序共存。 在 IA-32e 模式下运行的操作系统可以通过将其代码段描述符的 CS.L 位清除为 0 来执行现有的 16 位和 32 位应用程序。

  1. IA-32e 模式只允许 64 位操作系统进入。
  2. 操作系统基于代码段描述符的 L 位来决定是否启用 64 位模式。 64 位模式用来运行 64 位的程序，兼容模式可以运行 32 位/16 位程序。
  3. 64 位模式下，线性地址为 64 位，操作数默认是 32 位大小，但是可以使用 64 位硬编码前缀将其转换为 64 位操作数。
  4. 兼容模式允许传统的 16 位和 32 位应用程序在 64 位操作系统下运行，无需重新编译（但是不允许 Virtual 8086 模式应用程序或使用硬件任务管理运行）。 与 64 位模式一样，兼容模式由操作系统在单个代码段的基础上启用。 这意味着 64 位应用程序可以在处理器上运行（以 64 位模式），同时传统的 32 位应用程序（未针对 64 位重新编译）以兼容模式运行。

兼容模式类似于传统（legacy）的保护模式，应用程序只能访问前 4GB 线性地址空间，只能使用标准 IA-32 的指令和寄存器。但是兼容模式属于 IA-32e 模式下的子模式，需要遵循以下：

  1. 兼容模式下，虚拟 8086 模式、任务门、堆栈参数复制功能不可用。

  2. 在内核中，内核数据结构、地址转译、中断和异常处理使用 64-bit 模式机制而不是 32-bit 机制来处理。

    • 兼容模式下，线性到物理地址转换使用 64 位模式扩展页面转换机制。

    • 使用 64 位模式机制处理中断和异常。

    • 系统调用（通过调用门和 SYSENTER/SYSEXIT 调用）使用 IA-32e 模式机制处理。

兼容模式和 64-bit 模式是同时运行并存的，兼容模式应用程序不知道 64-bit 模式的存在，其根据代码段描述符的 L 位来进行运行。

在 IA-32e 模式下，所有 16 位和 32 位地址计算都进行零扩展以形成 64 位地址。 地址计算首先被截断为当前模式（64 位模式或兼容模式）的有效地址大小，被任何地址大小前缀覆盖。 结果是零扩展的完整 64 位地址宽度。 因此，在兼容模式下运行的 16 位和 32 位应用程序只能访问 64 位模式有效地址的低 4GBytes。 同样，在 64 位模式下生成的 32 位地址只能访问 64 位模式有效地址的低 4GBytes。

参考《64-Bit Extension Technology Software Developer’s Guide》。

2 CPUID 指令

CPUID 是一个指令，用来得到 CPU 处理器的身份信息（如型号、家族等）、特征和状态，还能得到 CPU 的一些其他扩展信息。

Opcode Instruction Op/En 64-Bit Mode Compat/Leg Mode Description
0F A2 CPUID ZO Valid Valid 根据输入的 EAX（在某些情况下，ECX 也作为输入）的不同，将不同的身份/特征信息返回到 EAX、EBX、ECX、EDX 中。

指令操作数：

Op/En Operand 1 Operand 2 Operand 3 Operand 4
ZO NA NA NA NA

2.1 CPUID 指令的支持

  1. CPUID 指令从 Intel 486 开始引入支持。

  2. 判断 CPU 是否支持 CPUID 指令：使用 RFLAGS 寄存器的 Bit-21（ID位）。

    在代码中，如果能设置或者清除 rflags.ID 位的值，则表示 CPU 支持 CPUID 指令。

  3. 在 IA-32e 模式下，CPUID 指令会清除 rax/rbx/rcx/rdx 的高 32 位，仅使用低 32 位。

2.2 CPUID 的使用

Intel 将要查询的功能号放入 RAX 寄存器中（对于较复杂的信息查询，会使用到 RCX 作为输入寄存器）。

输入分为两部分，主叶为必须值。

  • RAX：主叶，leaf 或 main-leaf
  • RCX：子叶，sub-leaf

CPUID 返回的信息分为两大类：基本信息、扩展信息。每类信息都有最大功能号（main）限制。

目前 Intel 手册上这两类信息的输入功能号范围：

  • 基本功能号：0～0x1F
  • 扩展功能号：0x80000000～0x80000008

  1. 查询最大基本功能号：使用 00h leaf 来查询。

  2. 查询最大扩展功能号：使用 80000000h leaf 来查询。

如果输入：功能号 <= 最大功能号，但是 CPU 不支持该功能号，则所有寄存器的返回值为 0。

如果输入：功能号 > 最大功能号，则返回 CPU 支持的最大功能号的信息。如上面实验 CPU 的 0x16/0x80000008 的信息。

除此之外还有几个常用的指令：

  1. 查询处理器型号（Family, model, stepping）：CPUID.01H:EAX。

  2. 最大的物理地址和线性地址：CPUID.80000008H:EAX[bits 7-0:Physical,bits 15-8:Linear]。

  3. 查询 MONOTOR/MWAIT 信息：CPUID.01H:ECX.[3]。

    monitor/mwait 指令的使用情形有些与 pause 指令类似，在 Idle-Loop（空闲的循环）中用来提高处理器的性能和减少能耗。pause 常用于 OS 的自旋锁，monitor/mwait 对某个地址范围做监控，这个地址范围内没发生写操作就进入优化（空闲）的状态，由 mwait 等待某些事件而退出，因地址 store 操作而退出或者因某些中断类的事件发生。由 IA32_MISC_ENABLE 开启这个功能。

  4. 查看 syscall/sysret 指令是否可用：CPUID.80000001H:EDX.[11] = 1。

3 CRx 寄存器

控制着 CPU 处理器的工作模式，限制着 CPU 的功能。

IA-32e 模式下 CR0～CR15 控制寄存器被扩展为 64 位，但是仅使用 CR0、CR2、CR3、CR4，还有专用于 IA-32e 模式下的 CR8。

  • CR0：包含系统控制位、操作模式位、处理器状态位。
  • CR1：保留。
  • CR2：发生页面异常（Page-fault）时，用来存放页面异常的线性地址。注意：mov cr2,xx 指令并不会检查地址 xx 是否规范。
  • CR3：唯一存放物理地址的寄存器。保存的物理地址为 PML4/PML5 表基址。
  • CR4：提供一些扩展的控制功能。
  • CR8：仅在 IA-32e 模式下可用。控制着对 Task Priority Register (TPR) 的读写。

注意：通用寄存器的访问和写入仅在内核模式下才可以使用。

3.2 CR2

CR2 没有控制位，当页面发生异常（缺页、页面访问权限错误）是，用来存放引发的 #PF 异常线性地址。

注意：mov cr2,xx 指令并不会检查地址 xx 是否规范。

3.3 CR3

CR3 寄存器中的值为物理地址。低 12 位中仅有两个标志位 PCD、PWT。

CR3 的值指向 PML4 或 PML5 时需要将低 12 位置 0，然后 CR3 的 64 位指向的物理地址就是 PML4 或 PML5 表的基地址。

注意：在使用 CR3 寻找 PML4 或 PML5 时，低 12 位需要置 0，所以存放 PML4 或 PML5 的页面必须与上一个页面 4KB 对齐。

PCD 和 PWT 标志控制处理器内部数据缓存中该分页结构的缓存（它们不控制页面目录信息的TLB缓存）。

  • PCD（Page-level Cache Disable）：页级缓存禁用。控制用于访问当前分页结构层次的第一个分页结构的内存类型。如果禁用分页、或使用 PAE 分页、或者在 CR4.PCIDE = 1 时使用 4 级分页或 5 级分页，则不使用该位。
  • PWT（Page-level Write-Through）：页级直写。控制用于访问当前分页结构层次的第一个分页结构的内存类型。如果禁用分页、或使用 PAE 分页、或者在 CR4.PCIDE = 1 时使用 4 级分页或 5 级分页，则不使用该位。

PWT：Page Write Through

  • PWT = 1，写Cache的时候也要将数据写入内存中。
  • PWT = 0，写Cache时由CPU控制寄存器决定是否写入内存。

PCD：Page Cache Disable

  • PCD = 1，禁止某个页写入缓存，直接写内存。
  • PCD = 0，可以写入Cache。

3.4 CR4

CR4 目前（2022.4）共 23 个标志位，这些标志可以实现一些架构扩展，并表明操作系统或对特定处理器功能的执行支持。在 CR4 中，部分的控制位必须要通过 CPUID.EAX=01H 查询是否获得支持。

标志 位 描述
VME, PVI 0, 1 VME：虚拟 8086 模式扩展（Virtual-8086 Mode Extensions），置 1 时启用虚拟 8086 模式下的中断和异常处理扩展。还提供了对虚拟中断标志（VIF）的硬件支持。
PVI：保护模式虚拟中断（Protected Mode Virtual Interrupts），置 1 启用保护模式下对虚拟中断标志（VIF）的硬件支持，置 0 时禁用保护模式下的 VIF 标志。
TSD 2 禁用时间戳指令（Time Stamp Disable）
TSD = 1，将 RDTSC 指令的执行限制 R0 运行的程序中。
TSD = 0，允许 RDTSC 指令在任何权限级别下执行。
类似于 CPUID.80000001H:EDX[27] = 1：RDTSCP and IA32_TSC_AUX are available。CPUID 查询 leaf 中有两个与 TSD ( Time Stamp Disable）相关：CPUID.EAX-01H 中 返回的 EDX[4].TSC 标志位用来测试是否支持 CR4.TSD 控制位：CPUID.EAX-80000001H 中返回的 EDX[27].RDTSCP 位用来测试是否支持 RDTSCP 指令。
DE 3 调试扩展（Debugging Extensions）
DE = 1，引用 DR4 和 DR5 时，由于 DR4 和 DR5 是被保留的，会产生一个 #UD 异常。
DE = 0，处理器会对寄存器 DR4 和 DR5 的引用进行别名，DR4 和 DR5 被映射到 DR6 和 DR7。以便于写在早期 IA-32 处理器上运行的软件兼容。
PSE 4 PSE：分页扩展（Page Size Extension），仅适用于 32 位的分页模式中。PDE.PS 位受 CR.PSE 位影响：
PAE 5 PAE：物理地址扩展（Physical Address Extension)
IA-32：PAE = 1，启用 2-9-9-12 分页，使分页产生超过 32 位的物理地址。PAE = 0，使用 10-10-12 分页，使用 32 位的物理地址。
IA-32e：在进入 IA-32e 模式之前必须设置 PAE。
MCE 6 机器异常检查（Machine-Check Exception）
置 1 时开启 Machine-Check 机制，清 0 时关闭 Machine-Check 机制。
CR4.DE 和 CR4.MCE 功能需要处理器支持，可以从 CPUID.EAX=01H.EDX 寄存器里获得，返回的 EDX[2]是 DE 支持位，EDX[7]是 MCE 支持位。
PGE 7 全局分页使能（Page Global Enable），该位影响 TLB 的刷新。启用该位前必须保证 CR0.PE = 1, CR0.PG = 1，即必须先启用保护模式和分页模式。
PGE = 1，PDE.G 和 PTE.G 才被启用。(CR4.PGE = 1) && (PDE.G = 1 \|\| PTE.G = 1) 条件下进程 CR3 切换时不会刷新 TLB。
PGE = 0，PDE.G 和 PTE.G 被禁用。
PCE 8 性能监控计数使能（Performance-Monitoring Counter Enable ）位
PCE = 1，RDPMC 指令可以在任何权限下执行。
PCE = 0，RDPMC 指令只能在 R0 级权限下执行。
这和 CR4.TSD 位的用法是相反的。说明 CR4.PCE 比 CR4.TSD 需要更谨慎地去使用。
OSFXSR, OSXMMEXCPT 9, 10 OSFXSR：和 x87 浮点计算相关
OSFXSR = 1，指示 SSE 系列指令是可用的，FXSAVE 和 FXRSTOR 指令可以保存 XMM 寄存器和 MXCSR 的状态。
OSFXSR = 0，FXSAVE 和 FXRSTOR 指令只能保存 x87 和 MMX 寄存器的状态，使用 SSE 系列指令会产生 #UD 异常。
CR4.OSFXSR 功能需要处理器支持，可以从 CPUID.EAX=01H.EDX 寄存器里获得，返回的 EDX[24] 是 FXSR 支持位。
OSXMMEXCPT：表示操作系统支持通过在 SIMD 浮点异常时调用的异常处理程序来处理未屏蔽 SIMD 浮点异常（# XM) 生成。 SIMD 浮点异常仅由 SSE/SSE2/SSE3/SSE4.1 SIMD 浮点指令生成。
UMIP 11 UMIP = 1，禁止在 CPL>0 时使用 SGDT, SIDT, SLDT, SMSW, 和STR 指令。
LA57 12 仅在 IA-32e模式下使用（57-bit linear addresses）
LA57 = 1，启用 5 级分页模式转换 57 位线性地址。
LA57 = 0，启用 4 级分页模式转换 48 位线性地址。
VMXE 13 启用 VT 模式
VMXE = 1，开启 VMX（Virtual Machine Extensions）功能，由 VMXON 指令进入VMX 模式，VMXOFF 指令退出 VMX 模式。
VMXE = 0，关闭 VMX 功能。将 CR4.VMXE 清 0 只能在VMX模式以外。
CR4.VMXE 需要处理器支持，可检查从 CPUID.EAX=01H 返回的 ECXI[5].VMX 看是否得到支持。
SMXE 14 SMXE = 1，表示开启 SMX（Safer Mode Extensions）功 能（”上帝模式“）。可以使用 GETSEC 指令查询 SMX 功能。
VMXE 与 SMXE 控制位仅在 Intel 上使用，AMD 机器上是保留位。
FSGSBASE 16 FSGSBASE = 1，启用 RDFSBASE, RDGSBASE, WRFSBASE, WRGSBASE指令。
PCIDE 17 PCIDE = 1，启用 Process-context identifiers (PCIDs)。仅适用于 IA-32e 模式（如果 IA32_EFER.LMA = 1）。
OSXSAVE 18 和 x87 FPU 的 XSAVE/XRSTOR 指令以及 XSETBV/XGETBV 指令有关。
KL 19 Key-Locker-Enable Bit
KL = 1，表示启用 LOADIWKEY 指令。如果系统固件已激活对 AES Key Locker 指令的支持，此时使用 CPUID 指令查询结果为 CPUID.19H:EBX.AESKLE[bit0] = 1 ，就表示已启用 AES Key Locker 指令。
KL = 0，此时使用 CPUID 指令查询结果为 CPUID.19H:EBX.AESKLE[bit0] = 0 ，此时执行任何 Key Locker指令会导致 #UD 异常。
SMEP 20 Supervisor Mode Executions Protection Enable
SMEP = 1，在 R0 权限的程序不可以执行 R3 中 PTE.XD = 0 的页面上的代码。
SMEP = 0，在 R0 权限的程序可以执行 R3 中 PTE.XD = 0 的页面上的代码。
SMAP 21 Supervisor Mode Access Protection Enable
SMEP = 1，在 R0 权限的程序不可以访问 R3 中 PTE.XD = 0 的页面上的代码。
SMEP = 0，在 R0 权限的程序可以访问 R3 中 PTE.XD = 0 的页面上的代码。
PKE, PKS 22, 24 PKE（Enable protection keys for user-mode pages）。4 级分页和 5 级分页将每个用户模式的线性地址与一个保护密钥联系起来。使用 PKRU 寄存器可以指定每个保护密钥关联的用户模式线性地址是否可以被读取或写入。可以通过 CPUID.(EAX=07H,ECX=0H):ECX.OSPKE[bit4]) 查看改位是否已置 1。
PKE 还控制是否允许使用 RDPKRU 和 WRPKRU 指令来访问 PKRU 寄存器。

PKS（Enable protection keys for supervisor-mode pages）。4 级分页和 5 级分页将 R0 模式的线性地址与一个保护密钥联系起来。PKS = 1 时，可以使用 MSR 的 IA32_PKRS 可以指定每个保护密钥关联的内核模式线性地址是否可以被读取或写入。
CET 23 控制流强制技术（Control-flow Enforcement Technology ）
CET = 1，启用控制流强制技术，只有在 CR0.Wp = 1 时才可以将 CET = 1。
必须在 CR0.WP 被清零之前将 CET = 0。

4 分段架构

4.1 IA-32 分段架构

IA-32 架构中，内存管理功能分为：段（Segment）、页（Page）。

段地址格式，也叫做逻辑地址或远指针（Far point）：Sengment-Selector: Offset（如 CS:Offset）。

如果不启用分页，如下图，处理器的线性地址将直接映射到处理器的物理地址空间。

IA-32 支持三种分段方式：

  1. 基础平坦模式（Basic Flat Model）

    这种模式下，至少存在两种段描述符：代码段描述符、数据段描述符（段基址都为 0，Limit 都为 4GB）。任何尝试访问不存在的内存时不会产生异常。

  2. 受保护的平坦模式（Protected Flat Model）

    定义了 R0 权限的代码段和数据段，以及 R3 权限的代码段和数据段。任何尝试访问不存在的内存时都会生成通用保护异常 (#GP)。

  3. 多分段模式（Multi-Segment Model）

    Windows 使用该分段机制，访问段内存时会执行段权限检查。

4.2 IA-32e 分段架构

IA-32e 模式下，分段的效果取决于处理器是以兼容模式（Compatibility-mode）还是 64 位模式（64-bit mode）运行。

  • 兼容模式：不会禁用，依然存在分段，分段的方式和 IA-32 下方式一样（请注意并不是直接使用 IA-32 的分段）。
  • 64 位模式：分段通常被禁用（但不是完全禁用），从而创建一个平坦的 64 位线性地址空间。处理器将CS、DS、ES、SS的段基址视为零，创建一个等于有效地址的线性地址。FS 段和 GS 段基址不为 0。处理器在 64 位模式下运行时不执行段Limit检查。由于 64 位模式下的平坦地址空间，线性地址等于有效地址。 如果 FS 或 GS 段与非零段基址一起使用，则此规则不成立。

段基址（Segment.Base）有区别：

  • IA-32：段基址为 32 位。
  • IA-32e：段基址为 64 位（其实主要是兼容模式下使用）。

IA-32e 模式下将逻辑地址转换为线性地址的方式相同，如下图：

在 IA-32e 模式下的代码段描述符的 L 位用来标志一个代码段按何种模式来运行代码：

  • L = 1：64 位代码段。
  • L = 0：
    • D/B = 1：32 位代码段。
    • D/B = 0：16 位代码段。

4.3 物理地址空间

  1. IA-32 架构保护模式下：

    • 物理地址是一个平坦的地址空间，未进行分段。

    • 物理地址范围为 4GB，启用物理地址扩展（PAE）或页面大小扩展 (PSE-36) 功能后，物理地址空间为范围 64 GB。

  2. IA-32e 架构处理器：物理地址范围的大小在不同 CPU 上支持不同。大小由 CPUID.80000008H:EAX[bits 7-0] 指出，如 2.2 实验中我的电脑最大就是 36 位（64GB）。

4.4 段寄存器

CPU 处理器提供了 6 个段寄存器：CS、DS、ES、FS、GS、SS。

段寄存器分为可见部分（段选择子）和隐藏部分：

  • IA-32：共 96 位
    • 可见部分：16 位
    • 隐藏部分：80 位
  • IA-32e：共 128 位
    • 可见部分：16 位
    • 隐藏部分：112 位

变化是 IA-32e 下段基址变为 64 位。

注意：在段寄存器内 Limit 占 32 位，在段描述符内只有 20 位。

当G=0时，Limit的范围是0x00000000-0x000FFFFF，其中，段描述符的20位在低位，高位补0。
当G=1时，Limit的范围是0x00000FFF-0xFFFFFFFF，其中，段描述符的20位在高位，低位补F。

IA-32e 的 64 位模式下不使用 ES、DS 和 SS 段寄存器，所以它们在段描述符寄存器中的字段（Base、Limit和Attribute）被忽略。某些形式的段加载指令也是无效的（例如， LDS，POP ES）。 引用 ES、DS 或 SS 段的地址计算被视为段基址为零。

FS.base 和 GS.base 的隐藏描述符寄存器字段物理映射到 MSR，以便加载 64 位实现支持的所有地址位。

R3:CS = 33, DS = 2B, ES = 2B, FS = 53, GS = 2B, SS = 2B

R0:cs=0010 ss=0018 ds=002b es=002b fs=0053 gs=002b

4.5 段选择子

段选择子是一个 16 位数，该数决定了取 GDT 表中查哪一个条目数据。

注意：处理器不使用 GDT 的第一个条目，第一项是全 0 的值。 指向 GDT 第一项的段选择子（即 Index = 0 且 TI = 0）被称为“空段选择子” 。段寄存器加载空段选择子时，处理器不会产生异常寄存器（除了 CS 或 SS 段寄存器）。但是，当使用包含空选择子的段寄存器访问内存时，它会生成异常。空选择子可用于初始化未使用的段寄存器，但不能加载来访问内存。 使用空段选择子加载 CS 或 SS 寄存器会导致生成通用保护异常（#GP）。

5 段描述符

5.1 段描述符表

段描述符表（Segment Descriptor Tables）：是一个数组，每个元素为 8 字节，表中的每个元素叫做段描述符。一共有两种段描述符表：

  • 全局描述符表 GDT（Global Descriptor Table）
  • 局部描述符表 LDT（Local Descriptor Table）

另外还有两个系统段描述符 IDT、TR。

段描述符是 GDT 表和 LDT 表的元素表示，它为段寄存器提供隐藏的字段：Base、Limit、Attribute。（注意：在段寄存器内 Limit 占 32 位，在段描述符内只有 20 位。）

  • IA-32：GDTR 寄存器为 48 位，低 16 位表明 GDT 表的字节总数，高 32 位为 GDT 表在内存中的起始线性地址。

5.2 Limit、G 位

在逻辑地址的表示为：Seg.Selector:Offset。

IA-32 架构

在 IA-32 下 Offset 是 32 位，以及段寄存器隐藏部分的 Limit 也是 32 位，但是在段描述符中仅有 20 位。所以需要根据段描述符的 G、D/B、以及代码段的 E 位来将 20 位扩展成 32 位。

Limit：20-bits，用来指定段的大小范围。

  • G = 1，段的范围是 0x00000FFF～0xFFFFFFFF（4KB～4GB），以 4KB 大小为增量，增加到 32 位。

  • G = 0，段的范围是 0x00000000～0x000FFFFF（0～1MB），以 1Byte 大小为增量，此时只使用段描述符中的 20 位。

G 位只是指定了段的大小范围，但是 D/B 则指定了数据段最大上限（适用于向下扩展的数据段）：

  • D/B = 1，段的最大上限值为 4GB（0xFFFFFFFF）。
  • D/B = 0，段的最大上限值为 64KB（0x0000FFFF）。

注意：除了代码段 CS，其他 DS、ES、FS、GS、SS 都可以归为数据段。数据段是有方向的，即向下扩展（Expand-down）还是向上扩展（Expand-up）。IA-32 架构使用的是向下扩展。

向上扩展、向下扩展、D/B 位的关系：

逻辑地址的表示为：Seg.Selector:Offset。其中 offset 取值范围就是依赖于向上扩展还是向下扩展。

  • 向上扩展：0 <= Offset < Limit
  • 向下扩展：Limit+1 <= Offset < 0xFFFFF/0xFFFFFFFF（最大值取决于D/B位的值）

要使 Offset 取值范围尽可能的大，也就是可使用的内存尽可能的多：

  • 对于向上扩展，需要不断增大 Limit 的值，所以”向上扩展“因此而得名。
  • 对于向下扩展，需要不断减小 Limit 的值，所以”向下扩展“因此而得名。
