1 同步与互斥

  • 并发：单核或多核下，在操作系统中，同个处理机上有多个程序同时运行即并发。并发可分为同步和互斥。
    • 同步：并发执行的环境中各个线程必须严格按照规定的某种先后次序来运行。
    • 互斥：同一个资源同一时间只有一个访问者可以进行访问，其他访问者需要等前一个访问者访问结束才可以开始访问该资源。但互斥无法限制访问者对资源的访问顺序，即访问是无序的。
  • 并行：仅在多核下，两个及以上数量的线程同一时刻在执行。

多核并发情况下，存在并行的情况。多线程同时处理某些资源就会存在出错的情况，而本文就主要探讨一下问题和解决。

注意⚠️：如果线程中的代码只对局部变量进行访问，就不存在并发的问题；只有当访问全局变量的时候，才需要关注并发的问题。这是因为局部变量每次访问都是在本次调用的堆栈里操作，不会产生互相影响。

1.4 高并发HOOK

一、

这个问题的关键是，hook 后一般是 e8 / e9 后跟4字节，总共5字节，但没办法一次性改5个字节，可能改了第一个字节，正要改后4个字节时，别的线程进来访问这块内存了，就会出错。

这里介绍三种办法：

  • 短跳中转，短跳中转（2字节）是比较常用的，修改前2字节跳到某个方便HOOK的地方（长跳），不多做介绍。
  • 中断门，中断门也是只用改两个字节，需要先构造中断门，也不介绍。
  • 找一条一次性修改8字节的指令，这个指令就是cmpxchg8b。

cmpxchg8b mem64指令的工作如下：
比较 mem64 和 EDX:EAX
如果相等，那么把 ECX:EBX 存储到 mem64
如果不相等，那么把 mem64 存储到 EDX:EAX

我们要一次性改8字节内存，用的是相等的情况，先把要写入的内容放到 ECX:EBX ，然后调用 cmpxchg8b 指令即可。

2 线程等待与唤醒

临界区和自旋锁，这两种同步方案在线程无法进入临界区时都会让当前线程进入等待状态。
一种是通过Sleep函数实现的，一种是通过让当前的CPU“空转”实现的，但这两种等待方式都有局限性：

  1. 通过Sleep函数进行等待，等待时间该如何确定呢（怎么确认多长时间后自己可以进入临界区）？
  2. 通过“空转”的方式进行等待，只有等待时间很短的情况下才有意义，否则对CPU资源是种浪费，并且自旋锁只能在多核的环境下才有意义。

有没有更加合理的等待方式呢？只有在条件成熟的时候才将当前线程唤醒？Windows在Ntdll!RtlEnterCriticalSection函数中其实就给出了一种折中的方式（当前使用临界区的线程不是CurrentThreadId的话就会调用RtlpWaitForCriticalSection.ZwWaitForSingleObject挂起当前线程）。其他情况就会自旋（可参考base/ntdll/ia64/critsect.s）。可参考：深入理解CRITICAL_SECTION。

六、线程如何存储等待块？

KTHREAD +0x070 WaitBlock : [4] _KWAIT_BLOCK是一个数组，当一个线程所等待的对象小于等于3个时并不会额外去申请内存存放等待块，会将被等待的对象的_KWAIT_BLOCK存到这个位置，其中第四个成员WaitBlock[3]只能用于存储计时器对象的等待块，其索引WaitKey == 0x102。

当线程所等待的对象小于等于3个时，WaitBlock[3]有以下两种状态：

  1. 如使用WaitSingleObject(HANDLE x, -1)，则WaitBlock[3].NextWaitBlock = NULL；
  2. 如使用WaitSingleObject(HANDLE x, y)，y > 0，则WaitBlock[3].NextWaitBlock 指向 WaitBlock[0]，然后WaitBlock[0].NextWaitBlock 指向 WaitBlock[3]。

当线程所等待的对象大于3个时，将会分配新的空间来存储等待块，并不会使用+0x070 WaitBlock来进行存储。

七、总结：
1、等待中的线程，一定在等待链表中（KiWaitListHead），同时也一定在这张网上（KTHREAD +5C的位置不为空）。
2、线程通过调用WaitForSingleObject/WaitForMultipleObjects函数将自己挂到这张网上。
3、线程什么时候会再次执行取决于其他线程何时调用相关函数，等待对象不同调用的函数也不同。

3 WaitForSingleObject分析

3.1 WaitForSingleObject

先分析一下WaitForSingleObject调用过程：

  1. NtWaitForSingleObject （base/ntos/ob/obwait.c）
    1）调用ObReferenceObjectByHandle函数，通过对象句柄找到等待对象结构体地址。
    2）调用KeWaitForSingleObject函数，进入关键循环。

  2. KeWaitForSingleObject：上半部分
    1） 向_KTHREAD(+70)位置的等待块赋值。
    2） 如果超时时间不为0，KTHREAD(+70)第四个等待块与第一个等待块关联起来：第一个等待块指向第四个等待块，第四个等待块指向第一个等待块。（_KWAIT_BLOCK第四个成员是给计时器用的）。
    3）KTHREAD(+5C)指向第一个_KWAIT_BLOCK。
    4） 进入关键循环。

总结：
不同的等待对象，用不同的方法来修改_DISPATCHER_HEADER（SignalState）。
比如：如果可等待对象是EVENT，其他线程通常使用SetEvent来设置SignalState = 1。
并且，将正在等待该对象的其他线程唤醒，也就是从等待链表（KiWaitListHead）中摘出来。但是，SetEvent函数并不会将线程从等待网上摘下来，是否要下来，由当前线程自己来决定。

3.2 NtWaitForSingleObject

该函数源码在base/ntos/ob/obwait.c中，函数大致功能如下。

  1. 先检查if(Timeout > 0 && PreviousMode != KernelMode)，更新Timeout。
  2. 使用参数一Handle调用ObReferenceObjectByHandle获得被等待对象Object。
  3. _OBJECT_HEADER.Type.DefaultObject = [[Object - 0x10] + 0x48]，得到DefaultObject，并对其进行判断，判断方法看下一节。
  4. 调用KeWaitForSingleObject。

⚠️需要注意参数二：参数Alertable是个布尔量，表示是否允许本次等待因用户空间APC而中断，或者说被“吵醒（Alert）”。

  • 吵醒与唤醒是不同的，唤醒是因为所等待的条件得到了满足（仓库到货），而吵醒是因为别的原因而醒来（与仓库无关）。
  • 举例，唤醒：等待时间到，吵醒：插入APC而去执行这个APC。

3.2.1 DefaultObject

可等待对象分为：直接可等待对象间接可等待对象。

  • 直接可等待对象：_DISPATCHER_HEADER结构为内核对象的第一个成员。
  • 间接可等待对象：内核对象第一个成员不是_DISPATCHER_HEADER结构，而是往该内核对象内嵌入一个直接可等待对象，所以实际可等待的是嵌入于其内部的事件对象。

_OBJECT_HEADER.Type.DefaultObject可以为指针或位移量（偏移）。

  • 指针：指向ObpDefaultObject，当前的对象并不是用来进行操作的直接等待对象，它指向一个默认的等待对象，然后传给KeWaitForSingleObject用，这种情况是特殊的，地址 > 0x80000000。（其实有点类似_FILE_OBJECT中并不是使用对象本身，而是使用加上偏移后的Event）。

  • 位移量：两种情况。

    • DefaultObject == 0，直接可等待对象就是对象本身。
    • DefaultObject > 0，直接可等待对象嵌入当前对象当中，如_FILE_OBJECT中嵌入Event。

LONG_PTR是个特殊的数据类型，它既可被用做指针，又可被用做整数。宏操作名IsPointerOffset()的意思是判断一个（形式上的）指针实际上是否是一个位移量。怎么判断呢？如果参数Ptr大于等于0，即其最高位为0，则就是一个位移量；而若小于0，即其最高位为1，则就是个指针。（实际上这里就是利用有符号数来进行判断，正数的最高位为1，同时内核地址肯定是大于0x80000000的，一举两得的方法）我们知道，在Windows中，系统空间与用户空间的分界是0x80000000，所以凡是指向内核中任何一个数据结构的指针，其最高位必定是1。所以，只要DefaultObject的最高位为0，就是位移量，而位移量为0就表示目标对象本身就是可等待对象。

为什么WaitObject要加上偏移？

NtWaitForSingleObject可以等待直接的、间接的可等待对象，然而KeWaitForSingleObject只能等待真正的可直接等待对象，所以，必须将间接可等待对象转换为直接可等待对象，而每个对象的可直接等待对象记录在其对象类型的DefaultObject字段中。

3.4 不同对象下修改SignalState

在分析 函数KeWaitForSingleObject时，当处理SignalState > 0情况时，针对互斥体_KMUTANT对象、事件_KEVENT对象、信号_KSEMAPHORE对象时，对SignalState的处理方式不同：

  • 互斥体（MutantObject）：SignalState -= 1。
  • 信号量（SemaphoreObject）：SignalState -= 1。

KiWaitTestWithoutSideEffects和KiWaitTestSynchronizationObject都是将在一个等待对象上所有的等待线程全部从等待网上摘出，并且清空将线程从等待链表中摘出。

前者和后者的区别：前者摘出链时SignalState不改变（为1），后者的WaitAny类型将SignalState置0。

以上函数中涉及到两个循环：

  1. 大循环：循环地将在该等待对象上的所有线程从等待网上摘掉，并且从线程等待链上摘掉。
  2. 小循环（KiUnwaitThread）：把一个线程上所有的被等待对象从自己的链上摘出，但是所有的等待块还在这个线程上挂着。

WaitEntry = ListHead->Flink;并没有错误，因为KiUnwaitThread摘链后，链表ListHead->Flink已经更新。

按我自己的理解KeSetEvent就应该根据被等待对象在不同线程上的等待方式WaitAny/WaitAny来设置if-else，而不是根据通知还是同步来设置：

  • Any：
    • 通知：将所有线程下等待网并唤醒
    • 同步：将等待该对象的第一个线程下等待网并唤醒（Set一次只能被用一次）
  • All：
    • 通知/同步：循环判断等待该对象的线程的其他被等待对象是否有信号，都有信号才将这个线程下网

源码中还有一个细节：在KeSetEvent中信号量设置为1后，调用KiWaitTest去将线程的等待块全部摘了，然后回到WaitForSingleObject中若判断SigleState不满足唤醒条件时，又回将等待块挂上去。但是还是不理解为什么要更改SigleState。

KeSetEvent：

  1. 如果等待队列是空的，没有线程在睡眠等待，就把Event->Header.SignalState设置成1。这样，如果此后有线程对此事件对象执行KetWaitForSingleObject()操作，就因此而不必睡眠等待。这对于通知型和同步型的事件对象都是一样。
  2. 已经有线程在这个对象上睡眠等待，那就要从中唤醒一个或所有线程。这时候的处理取决于事件对象的类型以及等待的方式：
    • 对于通知型的事件对象，或者等待者的等待方式是WaitAll，而且此前SignalState为0，就将SignalState置1，并通过KiWaitTest()唤醒这个线程，以及等待队列中所有符合条件的线程。可是，要是SignalState本来就已经是1，则没有任何影响。
    • 否则，对于同步型的事件对象，并且等待者的等待方式是WaitAny，就通过KiUnwaitThread()唤醒等待队列中的第一个线程。此时并不改变SignalState的值。
