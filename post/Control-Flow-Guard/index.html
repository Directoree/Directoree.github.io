1 CFG 支持

微软于 2014 年在 Windows 8.1 Update 3（KB3000850）中首次发布了 CFG（Control Flow Guard），并在之后的 Windows 10 中自动启用。（参考MSDNControl Flow Guard for platform security）

CFG 主要用来解决代码中针对间接跳转的缓解，如 jmp/call/ret 等间接跳转。代码运行过程中，在间接跳转之前，系统会通过 CFG Bitmap 来动态判断跳转目的地址是否是可信赖的区域。试图在汇编级别保护间接调用。这也叫做控制流完整性（Control Flow Integrity，CFI）。分为：

  • User CFG，在用户代码中的 CFG。
  • Kernel CFG，内核代码中的 KCFG。

CFG 的启用需要系统和编译器同时支持（就像ASLR一样），从 Virtual Studio 2015 Updated 2 版本以上开始支持（项目属性—C/C++—代码生成—控制流防护—是 guard:cf）。

2 CFG Bitmap

CFG 通过验证间接调用的目标是否位于已知函数的开头来解决 CFl 的间接 CALL/JMP 调用。如果目标不在已知函数的开头， 则该过程将简单地终止。

3 用户 CFG

参考PE metadata

  • 用户模式CFG只能对标记为地址空间布局随机化（ASLR）兼容的图像强制执行（由与Microsoft链接器的/DYNAMICBASE选项指定）。这是由于操作系统内部如何处理CFG，它基本上连接到ASLR基础设施。一般来说，作为第一步，CFG用户应该为他们的图像启用ASLR。工具不应假设操作系统在没有ASLR设置的情况下始终忽略CFG，而通常应该同时设置两者。
  • 请注意，要执行kCFG，需要虚拟化基础安全（VBS）。然而，即使VBS被禁用，kCFG也有一些有限的功能。这超出了这篇博客文章的范围）。

只读的CFG 位图由操作系统维护，部分位图由所有进程共享。CFG 位图中：

  • 偶数位对应一个 16 字节对齐的地址。
  • 奇数位对应 15 个非 16 字节对齐的地址。

Bitmap 的核心就是根据目标地址本身，就能判断该地址是否是有效的跳转地址。

因此，内存管理器将 CFG 位图视为页面文件支持的可共享内存区域，并且对应于共享位的物理页面仅在 RAM 中存在一次。

CFG位图构建：

  1. 系统初始化时，会调用 MiInitializeCfg 函数来初始化对 CFG 的支持。 该函数创建一个或两个内存区对象 (MmCreateSection) 作为保留内存，其大小适合平台，如前所示。 对于 32 位平台，一张位图就足够了。 对于 x64 平台，需要两个位图 — 一个用于 64 位进程，另一个用于 Wow64 进程（32 位应用程序）。 内存区对象的指针存储在 MiState 全局变量的子结构中。
  2. 创建进程后，相应的内存区将安全地映射到进程的地址空间中。 这里的安全意味着该部分不能被进程中运行的代码取消映射或更改其保护。 （否则，恶意代码可能只是取消映射内存、重新分配并用 1 位填充所有内容，从而有效地禁用 CFG，或者通过将区域标记为读/写来简单地修改任何位。）

用户模式 CFG 位图在以下场景中填充：

  1. 在镜像映射期间，由于 ASLR（有关 ASLR 的更多信息，请参阅第 5 章）而动态重新定位的镜像将提取其间接调用目标元数据。 如果镜像没有间接调用目标元数据，这意味着它不是用 CFG 编译的，则假设镜像中的每个地址都可以间接调用。 正如所解释的，因为动态重新定位的镜像应该在每个进程中加载到相同的地址，所以它们的元数据用于填充用于 CFG 位图的共享内存区。
  2. 在镜像映射期间，需要特别注意非动态重新定位的镜像和未映射到其首选基础的镜像。 对于这些镜像映射，CFG 位图的相关页面是私有的，并使用镜像中的 CFG 元数据进行填充。 对于其 CFG 位存在于共享 CFG 位图中的镜像，将进行检查以确保所有相关的 CFG 位图页面仍然共享。 如果不是这种情况，则使用镜像中的 CFG 元数据填充私有 CFG 位图页面的位。
  3. 当虚拟内存被分配或重新保护为可执行文件时，CFG 位图的相关页面默认设为私有并初始化为全 1。 这在即时编译等情况下是必需的，在这种情况下，代码会即时生成然后执行（例如，.NET 或 Java）。

尽管 CFG 足以防止利用间接调用或跳转的攻击类型，但可以通过以下方式绕过它：

  1. 如果进程可能被欺骗或现有的 JIT 引擎被滥用来分配可执行内存，则所有相应的位都将设置为 {1, 1}，这意味着所有内存都被视为有效的调用目标。
  2. 对于 32 位应用程序，如果预期的调用目标是 __stdcall（标准调用约定），但攻击者能够将间接调用目标更改为 __cdecl（C 调用约定），则堆栈将损坏，因为 C 调用函数将 与标准调用函数不同，不执行调用者参数的清理。 由于 CFG 无法区分不同的调用约定，这会导致堆栈损坏，可能具有攻击者控制的返回地址，从而绕过 CFG 缓解措施。
  3. 同样，编译器生成的 setjmp/longjmp 目标的行为与真正的间接调用不同。 CFG无法区分两者。
  4. 某些间接调用更难保护，例如导入地址表 (IAT) 或延迟加载地址表，它们通常位于可执行文件的只读部分。
  5. 导出的函数可能不是理想的间接函数调用。

Windows 10 引入了 CFG 的改进来解决所有这些问题。 第一个是向 VirtualAlloc 函数引入一个名为 PAGE_TARGETS_INVALID 的新标志，并向 VirtualProtect 引入一个名为 PAGE_TARGETS_NO_UPDATE 的新标志。 设置这些标志后，分配可执行内存的 JIT 引擎将不会看到所有分配的位都设置为 {1, 1} 状态。 相反，他们必须手动调用 SetProcessValidCallTargets 函数（调用本机 NtSetInformationVirtualMemory 函数），这将允许他们指定其 JIT 代码的实际函数起始地址。 此外，此函数被标记为带有 DECLSPEC_GUARD_SUPPRESS 的抑制调用，确保攻击者无法使用间接 CALL 或 JMP 重定向到它，即使是在函数启动时也是如此。 （因为它本质上是一个危险的函数，使用受控堆栈或寄存器调用它可能会导致绕过 CFG。）

接下来，改进的 CFG 将您在本节开头看到的默认流程更改为更精细的流程。 在这个流程中，加载器并没有实现一个简单的“验证目标，返回”功能，而是一个“验证目标，调用目标，检查堆栈，返回”功能，它被用在 32 位应用程序的一个子集地方（ 和/或在 Wow64 下运行）。这种改进的执行流程如图 7-32 所示。

接下来，改进后的 CFG 在可执行文件中添加了额外的表，例如 Address Taken IAT 表和 Long Jump Address 表。 当编译器启用longjmp和IAT CFG保护时，这些表用于存储这些特定类型的间接调用的目标地址，相关函数不放在常规函数表中，因此不在位图中。 这意味着如果代码试图间接跳转/调用这些函数之一，它将被视为非法转换。 相反，C 运行时和链接器将通过手动检查此表来验证 longjmp 函数的目标。 虽然它比位图效率低，但这些表中应该几乎没有功能，因此成本可以承受。

最后，改进的 CFG 实现了一个称为导出抑制的功能，该功能必须得到编译器的支持并通过进程缓解策略启用。 （有关进程级别缓解的更多信息，请参阅“进程缓解策略”部分。）启用此功能后，将实现一个新的位状态（回想一下，项目符号列表将 {0, 1} 作为未定义状态）。 此状态表示该函数有效但导出受抑制，加载程序将对其进行不同的处理。

您可以通过查看镜像加载配置目录中的保护标志来确定给定二进制文件中存在哪些功能，之前使用的 DumpBin 应用程序可以对其进行解码。 作为参考，它们列在表 7-21 中。

Loader 与 CFG 交互：

尽管构建 CFG 位图的是内存管理器，但用户模式加载器（有关更多信息，请参见第 3 章）有两个用途。 第一种是仅在启用该功能时才动态启用 CFG 支持（例如，调用者可能没有为子进程请求 CFG，或者进程本身可能没有 CFG 支持）。 这是由 LdrpCfgProcessLoadConfig 加载器函数完成的，该函数被调用来为每个加载的模块初始化 CFG。 如果PE可选头文件中的模块DllCharacteristics标志没有设置CFG标志（IMAGE_DLLCHARACTERISTICS_GUARD_CF），IMAGE_LOAD_CONFIG_DIRECTORY结构的GuardFlags成员没有设置IMAGE_GUARD_CF_INSTRUMENTED标志，或者内核已经强制关闭了这个模块的CFG，那么 没有什么可以做的。

其次，如果模块确实使用了 CFG，LdrpCfgProcessLoadConfig 会获取从映像中检索到的间接检查函数指针（IMAGE_LOAD_CONFIG_DIRECTORY 结构的 GuardCFCheckFunctionPointer 成员），并将其设置为 Ntdll 中的 LdrpValidateUserCallTarget 或 LdrpValidateUserCallTargetES，具体取决于是否启用了导出抑制。 此外，该函数首先确保间接指针没有以某种方式被修改为指向模块本身之外。

此外，如果使用改进的 CFG 来编译此二进制文件，则可以使用第二个间接例程，称为 dispatch CFG 例程。 它用于实现前面描述的增强执行流程。 如果镜像包含这样的函数指针（在上述结构的 GuardCFDispatchFunctionPointer 成员中），则将其初始化为 LdrpDispatchUserCallTarget，如果启用了导出抑制，则初始化为 LdrpDispatchUserCallTargetES。

提示：在某些情况下，内核本身可以代表用户模式模拟或执行间接跳转或调用。 在这种情况下，内核会实现自己的 MmValidateUserCallTarget 例程，该例程执行与 LdrpValidateUserCallTarget 相同的工作。

启用 CFG 时编译器生成的代码发出一个间接调用，该调用位于 Ntdll 中的 LdrpValidateCallTarget(ES) 或 LdrpDispatchUserCallTarget(ES) 函数中。 此函数使用目标分支地址并检查函数的位状态值：

  • 如果位状态为 {0, 0}，则分派可能无效。
  • 如果位状态为{1, 0}，且地址为 16 字节对齐，则分派有效。 否则，它可能无效。
  • 如果位状态为 {1, 1}，并且地址不是 16 字节对齐的，则分派有效。 否则，它可能无效。
  • 如果位状态为 {0, 1}，则分派可能无效。

如果分派可能无效，则 RtlpHandleInvalidUserCallTarget 函数将执行以确定适当的操作。 首先，它检查进程中是否允许被抑制的调用，这是一个不寻常的应用程序兼容性选项，如果启用了应用程序验证程序，或通过注册表可以设置。 如果是这样，它将检查地址是否被抑制，这就是为什么它没有被插入到位图中的原因（回想一下 guard 函数表条目中的 3 个特殊标志表明了这一点）。 如果是这种情况，则允许呼叫通过。 如果该函数根本无效（意味着它不在表中），则分派中止并且进程终止。

其次，检查是否启用了出口抑制。 如果是，则根据出口抑制地址列表检查目标地址，这再次用添加到保护功能表条目中的另一个标志指示。 如果是这种情况，加载程序会验证目标地址是对另一个 DLL 的导出表的转发器引用，这是对具有抑制导出的图像进行间接调用的唯一允许的情况。 这是通过复杂的检查来完成的，该检查确保目标地址位于不同的图像中，其图像加载目录已启用导出抑制，并且该地址位于该图像的导入目录中。 如果这些检查匹配，内核将通过前面描述的 NtSetInformationVirtualMemory 调用来调用，以将位状态更改为 {1, 0}。 如果这些检查中的任何一个失败，或者未启用导出抑制，则该过程将终止。

对于 32 位应用程序，如果为进程启用了 DEP，则会执行额外的检查。 （有关 DEP 的更多信息，请参阅第 5 章。）否则，因为没有执行保证，所以允许错误调用，因为它可能是一个较旧的应用程序出于合法原因调用堆或堆栈。

最后，由于未提交大量 {0, 0} 位状态以节省空间，如果检查 CFG 位图落在保留页上，则会发生访问冲突异常。 在 x86 上，异常 dll 设置很昂贵，而不是作为验证代码的一部分进行处理，而是让它正常传播。 （有关异常调度的更多信息，请参阅第 2 部分中的第 8 章。）用户模式调度程序处理程序 KiUserExceptionDispatcher 在验证函数中具有识别 CFG 位图访问违规异常的特定检查，如果异常代码是，它将自动恢复执行这简化了代码 在 LdrpValidatelserCallTarget(ES) 和 LdrpValidateUserCallTarget(ES) 和 LdrpDispatchUserCallTarget(ES) 中，它们不必包含异常处理代码。 在 x64 上，异常处理程序只是在表中注册，LdrpICallHandler 处理程序改为运行，其逻辑与上述相同。

Kernel CFG:

尽管使用 Visual Studio 和 /guard:cf 编译的驱动程序最终也具有与用户模式图像相同的二进制属性，但 Windows 10 的第一个版本并未对这些数据执行任何操作。与受保护的用户模式 CFG 位图不同 如果要创建一个更高、更可信的实体（内核），则没有任何东西可以真正“保护”内核 CFG 位图。 恶意利用可以简单地编辑与包含要修改的所需位的页面对应的 PTE，将其标记为读/写，然后继续进行间接调用或跳转。 因此，设置这种微不足道的可绕过缓解措施的开销根本不值得。

随着越来越多的用户启用 VBS 功能，可以再次利用 VTL 1 提供的更高安全边界。 SLAT 页表条目通过提供第二个边界来防止 PTE 页面保护更改来解决这个问题。 虽然位图对于 VTL 0 是可读的，因为 SLAT 条目被标记为只读，但如果内核攻击者试图更改 PTE 以将其标记为可读/可写，则他们无法对 SLAT 条目执行相同的操作。 因此，这将被检测为 HyperGuard 可以对其采取行动的无效 KCFG 位图访问（仅出于遥测原因——因为这些位无论如何都无法更改）。

KCFG 的实现几乎与常规 CFG 相同，除了未启用导出抑制、longjmp 支持以及为 JIT 目的动态请求额外位的能力之外。 内核驱动程序不应该做任何这些事情。 取而代之的是，这些位是根据“地址获取 IAT 表”条目（如果已设置）在位图中设置的； 每次加载驱动程序映像时，通过保护表中的常用函数条目； 以及由 MiInitializeKernelCfg 引导期间的 HAL 和内核。 如果未启用虚拟机管理程序，并且不存在 SLAT 支持，那么这些都不会被初始化，并且 Kernel CFG 将保持禁用状态。

就像在用户模式下一样，加载配置数据目录中的动态指针被更新，在启用的情况下，它将指向 __guard_check_icall 用于检查功能，而 __guard_dispatch_icall 用于增强 CFG 模式下的调度功能。 此外，名为 guard_icall_bitmap 的变量将保存位图的虚拟地址。

关于内核 CFG 的最后一个细节是，不幸的是，动态驱动程序验证程序设置将不可配置（有关驱动程序验证程序的更多信息，请参阅第 6 章，“/O 系统”），因为这需要添加动态内核挂钩并将执行重定向到 可能不在位图中的功能。在这种情况下，将返回 STATUS_VRF_CFG_ENABLED (0xC000049F)，并且需要重新启动（此时可以使用 Verifier 驱动程序挂钩构建位图）。
